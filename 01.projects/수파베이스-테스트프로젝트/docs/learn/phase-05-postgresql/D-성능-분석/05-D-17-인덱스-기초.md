## 이 파일에서 배우는 것

- **인덱스의 개념**: 데이터베이스가 데이터를 빠르게 찾는 방법
- **주요 인덱스 종류**: B-tree, HASH, GiST 등 (B-tree가 가장 흔함)
- **인덱스 생성 및 삭제**: `CREATE INDEX`, `DROP INDEX` 문법

- **비용-이득 분석**: 인덱스의 유지 비용과 성능 이득

## 인덱스 = 책의 목차

### FE 개발자를 위한 비유

책을 생각해보세요.
- **인덱스 없음 (Sequential Scan)**: 책 첫 장부터 끝까지 한 장씩 넘기며 찾고 싶은 내용을 찾는 것
  - 1000장 책에서 800장에 있는 내용을 찾으면 799장을 읽어야 함
  - **시간**: O(n)

- **인덱스 있음 (Index Scan)**: 책 뒤의 색인(index)을 열어서 원하는 내용의 페이지 번호를 찾고 바로 그 페이지로 이동
  - 색인은 알파벳 순서로 정렬되어 있어서 빠르게 찾을 수 있음
  - **시간**: O(log n)

PostgreSQL도 똑같습니다. 인덱스는 **정렬된 룩업 테이블**이어서 데이터베이스가 원하는 행을 빠르게 찾을 수 있게 해줍니다.

### 또 다른 비유: React의 useMemo

```javascript
// 인덱스 없는 쿼리 = 매번 전체 리스트를 순회
const filteredUsers = users.filter(u => u.id === targetId);

// 인덱스 있는 쿼리 = 미리 계산된 맵에서 조회
const userMap = useMemo(() => {
  const map = {};
  users.forEach(u => { map[u.id] = u; });
  return map;
}, [users]);
const user = userMap[targetId]; // O(1) 조회
```

인덱스는 데이터베이스의 "성능 최적화 미리계산" 입니다.

## 인덱스가 필요한 이유: Seq Scan vs Index Scan

### 실제 성능 차이를 EXPLAIN으로 보기

1000만 개 행이 있는 `users` 테이블에서 특정 사용자를 찾는 경우:

**인덱스 없음:**
```sql
EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'john@example.com';

-- 결과:
-- Seq Scan on users  (cost=0.00..200000.00 rows=1)
-- Filter: (email = 'john@example.com')
-- Actual rows: 1
```
- **cost**: 200000 (전체 테이블을 읽어야 함)
- **실행 시간**: ~1-2초

**인덱스 있음:**
```sql
CREATE INDEX idx_users_email ON users(email);

EXPLAIN ANALYZE
SELECT * FROM users WHERE email = 'john@example.com';

-- 결과:
-- Index Scan using idx_users_email on users  (cost=0.29..8.30 rows=1)
-- Index Cond: (email = 'john@example.com')
-- Actual rows: 1
```
- **cost**: 8.30 (인덱스를 통한 직접 접근)
- **실행 시간**: ~1ms (1000배 빠름!)

### 규칙
- **WHERE 조건으로 자주 사용되는 컬럼** → 인덱스하면 Seq Scan이 Index Scan으로 바뀜
- 비용이 크게 줄어듦 = 쿼리 응답 속도 빠름

## B-tree 인덱스 (기본 및 가장 흔한 종류)

### B-tree란?

**Balanced tree** (균형잡힌 트리)라는 뜻입니다. 데이터를 정렬된 상태로 저장하고, 이진 검색(binary search) 같은 방식으로 빠르게 데이터를 찾을 수 있습니다.

```
데이터:  5, 2, 8, 1, 9, 3, 7

B-tree 구조 (깊이 2):
           [5]
         /     \
      [2,3]   [7,8,9]
      /  |  \   /  |  \
    [1] [2] [3] [7] [8] [9]
```

각 노드에서 이진 검색으로 빠르게 다음 노드를 결정하므로, n개의 데이터를 찾는 데 log₂(n) 번의 비교만 필요합니다.

### PostgreSQL에서 기본 인덱스 = B-tree

명시적으로 타입을 지정하지 않으면 B-tree를 생성합니다.

```sql
-- B-tree 인덱스 (명시적)
CREATE INDEX idx_users_id ON users(id);
CREATE INDEX idx_users_id ON users USING BTREE(id);

-- 둘 다 같음 (기본값이 BTREE)
```

### B-tree가 지원하는 연산

```sql
-- 이 모든 연산이 B-tree 인덱스로 빨라집니다

-- 정확한 매칭 (=)
SELECT * FROM users WHERE id = 123;

-- 범위 검색 (<, >, <=, >=)
SELECT * FROM users WHERE created_at >= '2026-01-01';

-- 범위 (BETWEEN)
SELECT * FROM orders WHERE amount BETWEEN 100 AND 1000;

-- 정렬 (ORDER BY) — 인덱스가 이미 정렬되어 있으면 빠름
SELECT * FROM users ORDER BY created_at DESC;

-- IN 연산
SELECT * FROM users WHERE status IN ('active', 'pending');
```

### 다른 인덱스 타입 (간단한 소개)

| 타입 | 용도 | 예 |
|------|------|-----|
| **BTREE** | 일반 데이터 (숫자, 텍스트, 날짜) | email, user_id, created_at |
| **HASH** | 정확한 매칭만 필요할 때 | 많은 행을 정확히 찾을 때 |
| **GiST** | 기하학적 데이터 (위치, 도형) | PostGIS 확장 (좌표, 거리) |
| **GIN** | 배열, 전문검색 | 태그 배열, 텍스트 검색 |

FE 개발자가 Supabase에서 자주 쓰는 것은 **BTREE**와 **GIN** 정도입니다.

## 인덱스 생성 및 삭제

### 생성: CREATE INDEX

```sql
-- 기본 형태
CREATE INDEX idx_users_email ON users(email);

-- 유니크 인덱스 (중복 불가)
-- NOT NULL 제약이 함께 적용됨
CREATE UNIQUE INDEX idx_users_email_unique ON users(email);

-- 오름차순 (기본값)
CREATE INDEX idx_users_created ON users(created_at ASC);

-- 내림차순
CREATE INDEX idx_users_created_desc ON users(created_at DESC);
```

### 삭제: DROP INDEX

```sql
-- 인덱스 삭제
DROP INDEX idx_users_email;

-- 존재하지 않으면 에러 → 안전하게 삭제
DROP INDEX IF EXISTS idx_users_email;

-- CASCADE: 인덱스를 사용하는 다른 객체도 삭제
-- (일반적으로는 필요 없음)
DROP INDEX IF EXISTS idx_users_email CASCADE;
```

### 인덱스 목록 확인

```sql
-- 현재 테이블의 모든 인덱스 확인
SELECT schemaname, tablename, indexname, indexdef
FROM pg_indexes
WHERE tablename = 'users'
ORDER BY indexname;

-- 또는 \d 명령어 (psql)
\d users
```

## 복합 인덱스 (Multi-column Index)

### 언제 필요한가?

여러 컬럼의 조합으로 자주 검색할 때:

```sql
-- 예: user_id와 created_at 두 조건을 함께 사용
SELECT * FROM orders
WHERE user_id = 42 AND created_at >= '2026-01-01';
```

이 쿼리를 최적화하려면:

```sql
-- 좋음: 두 컬럼이 함께 정렬된 인덱스
CREATE INDEX idx_orders_user_created ON orders(user_id, created_at);
```

### B-tree 복합 인덱스의 작동 원리

```
인덱스: (user_id, created_at)

구조:
[user_id=1]
├─ 2024-01-01
├─ 2024-01-05
└─ 2024-01-10
[user_id=2]
├─ 2024-02-01
├─ 2024-02-15
└─ 2024-03-01
```

- **첫 번째 컬럼(user_id)**: 전체적으로 정렬됨
- **두 번째 컬럼(created_at)**: 각 user_id 그룹 내에서 정렬됨

### 쿼리별 효과

```sql
-- ✅ 인덱스 완벽 활용 (양쪽 컬럼 모두 검색)
SELECT * FROM orders
WHERE user_id = 42 AND created_at >= '2026-01-01';

-- ✅ 첫 번째 컬럼으로 활용 (user_id만 검색)
SELECT * FROM orders
WHERE user_id = 42;

-- ⚠️ 두 번째 컬럼만 검색 (인덱스 거의 안 씀)
-- 첫 번째 컬럼 없이 두 번째 컬럼만 검색하면 효율 떨어짐
SELECT * FROM orders
WHERE created_at >= '2026-01-01';

-- ❌ 전혀 다른 컬럼 (인덱스 미활용)
SELECT * FROM orders
WHERE amount > 100;
```

### 순서가 중요함

```sql
-- 좋음: (user_id, created_at) 순서대로
CREATE INDEX idx_orders_user_created ON orders(user_id, created_at);

-- 덜 좋음: (created_at, user_id) 순서라면
-- created_at 없이 user_id만 검색할 때 인덱스 미활용
CREATE INDEX idx_orders_created_user ON orders(created_at, user_id);
```

### 일반 규칙

**복합 인덱스를 설계할 때:**
1. **WHERE 조건에서 자주 쓰는 순서대로** 배열
2. **동등 비교(=) 컬럼들을 앞에**, 범위 비교(<, >=) 컬럼을 뒤에
3. 3개 이상 컬럼은 신중하게 (유지 비용 ↑)

## 부분 인덱스 (Partial Index)

### 언제 필요한가?

전체 테이블이 아닌 **일부 행만** 자주 검색할 때:

```sql
-- 예: 활성 사용자만 자주 검색
SELECT * FROM users WHERE status = 'active' AND email = 'john@example.com';

-- 문제: 전체 사용자 중 90%가 'deleted' 상태라면,
-- 일반 인덱스는 필요 없는 데이터까지 저장
```

### 부분 인덱스로 최적화

```sql
-- 활성 사용자(status = 'active')만 인덱싱
CREATE INDEX idx_active_users_email
ON users(email)
WHERE status = 'active';

-- 이 쿼리는 인덱스를 사용
SELECT * FROM users
WHERE status = 'active' AND email = 'john@example.com';

-- 이 쿼리는 인덱스를 사용 안 함
-- (status 조건이 WHERE에 없음)
SELECT * FROM users
WHERE email = 'john@example.com';
```

### 부분 인덱스의 이점

- **저장 공간 절약**: 인덱스 크기 줄어듦
- **쓰기 성능**: 활성 행에만 인덱스 업데이트 필요
- **유지 비용 감소**: VACUUM, ANALYZE 시간 단축

### 실전 예제

```sql
-- soft delete를 사용하는 경우
-- 삭제되지 않은 행만 인덱싱
CREATE INDEX idx_orders_user_deleted_at
ON orders(user_id)
WHERE deleted_at IS NULL;

-- 활성 구독만 인덱싱
CREATE INDEX idx_subscriptions_active
ON subscriptions(user_id)
WHERE active = true;
```

## 인덱스의 비용: 대가를 이해하자

### 인덱스가 성능을 떨어뜨리는 경우

#### 1. 쓰기(INSERT, UPDATE, DELETE) 성능 저하

```sql
-- 인덱스 1개: INSERT 약 1.1배 느림
CREATE INDEX idx_users_email ON users(email);
INSERT INTO users(id, name, email) VALUES(1, 'John', 'john@example.com');
-- DB: 행 저장 + 인덱스 업데이트 (2배 작업)

-- 인덱스 5개: INSERT 약 1.5배 느림
CREATE INDEX idx1 ON users(email);
CREATE INDEX idx2 ON users(created_at);
CREATE INDEX idx3 ON users(status);
CREATE INDEX idx4 ON users(user_id);
CREATE INDEX idx5 ON users(name);
-- DB: 행 저장 + 5개 인덱스 모두 업데이트
```

**규칙**: 인덱스가 많을수록 쓰기 작업이 느려집니다.

#### 2. 디스크 사용량 증가

```sql
-- 1000만 행 테이블
CREATE TABLE users(
  id BIGINT,
  email TEXT,
  name TEXT,
  created_at TIMESTAMP,
  status TEXT
);

-- 인덱스 없음: 약 2GB
-- 인덱스 3개: 약 3.5GB (1.5GB 증가)
-- 인덱스 10개: 약 6GB (4GB 증가)
```

**규칙**: 인덱스는 저장 공간도 차지합니다.

#### 3. 메모리 사용량

PostgreSQL은 자주 접근하는 인덱스를 메모리에 캐시합니다. 인덱스가 크면 메모리 압박 가능.

### 비용-이득 분석

```
쓰기가 많은 경우          |  읽기가 많은 경우
- 주문 실시간 입력        |  - 사용자 프로필 조회
- 로그 테이블            |  - 검색 쿼리
- 센서 데이터            |  - 필터링된 목록

인덱스는 2-3개만        |  인덱스 5-10개 고려
읽기 성능 < 쓰기 성능     |  읽기 성능이 더 중요
```

## 인덱스를 만들어야 할 때 / 피해야 할 때

### ✅ 인덱스를 생성하세요

#### 1. SELECT에서 자주 사용되는 WHERE 조건

```sql
-- 자주 실행되는 쿼리
SELECT * FROM users WHERE email = ?;

-- 인덱스 생성
CREATE INDEX idx_users_email ON users(email);
```

#### 2. JOIN 조건

```sql
-- orders.user_id로 자주 JOIN
SELECT o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id;

-- users.id는 PRIMARY KEY라 자동 인덱싱됨
-- orders.user_id에는 인덱스 추가
CREATE INDEX idx_orders_user_id ON orders(user_id);
```

#### 3. ORDER BY 정렬

```sql
-- 최신순 정렬이 자주 필요
SELECT * FROM posts
ORDER BY created_at DESC;

-- 인덱스 추가 (정렬 성능 향상)
CREATE INDEX idx_posts_created_desc ON posts(created_at DESC);
```

#### 4. DISTINCT, GROUP BY 대상

```sql
-- 카테고리별 집계
SELECT category, COUNT(*)
FROM products
GROUP BY category;

-- category에 인덱스
CREATE INDEX idx_products_category ON products(category);
```

#### 5. 범위 검색

```sql
-- 가격 범위 검색
SELECT * FROM products WHERE price BETWEEN 100 AND 1000;

-- 범위 인덱스
CREATE INDEX idx_products_price ON products(price);
```

### ❌ 인덱스를 피하세요

#### 1. 카디널리티가 매우 낮은 컬럼

```sql
-- 나쁨: boolean 컬럼 (값이 2개뿐: true, false)
CREATE INDEX idx_users_is_active ON users(is_active);
-- 인덱스 사용 이득: 거의 없음 (결국 절반 이상의 행 읽어야 함)

-- 나쁨: 성별 (M, F)
CREATE INDEX idx_users_gender ON users(gender);

-- 개선: 부분 인덱스 사용
CREATE INDEX idx_active_users ON users(id) WHERE is_active = true;
```

**규칙**: 값의 다양성이 낮으면 인덱스 효과 미미.

#### 2. 자주 업데이트되는 컬럼

```sql
-- 나쁨: 조회수 (매번 +1로 계속 변함)
CREATE INDEX idx_products_view_count ON products(view_count);
-- 영향: 모든 UPDATE 시 인덱스도 갱신 필요

-- 개선: 조회수는 인덱싱 말고, 캐시 사용 (Redis)
```

#### 3. 거의 쿼리되지 않는 컬럼

```sql
-- 나쁨: 생성 후 절대 검색 안 하는 컬럼
CREATE INDEX idx_users_metadata ON users(metadata);

-- 확인: 인덱스 사용 통계
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0;  -- 0번 사용됨
```

#### 4. 작은 테이블

```sql
-- 나쁨: 10-100개 행 테이블 (sequential scan이 더 빠를 수 있음)
CREATE INDEX idx_regions_code ON regions(code);
-- 인덱스 오버헤드가 이득보다 클 수 있음
```

**규칙**: 테이블이 작으면 인덱싱 효과 미미.

## Supabase에서 자주 사용하는 인덱스 패턴

### 패턴 1: User ID 필터링

```sql
-- 자주 하는 쿼리: 특정 사용자의 모든 데이터
SELECT * FROM posts WHERE user_id = ?;

CREATE INDEX idx_posts_user_id ON posts(user_id);

-- 복합: user_id + created_at (최신순 정렬도 빠르게)
CREATE INDEX idx_posts_user_created ON posts(user_id, created_at DESC);
```

### 패턴 2: 시간 범위 검색

```sql
-- 자주 하는 쿼리: 최근 N일 데이터
SELECT * FROM events
WHERE created_at >= NOW() - INTERVAL '7 days'
ORDER BY created_at DESC;

CREATE INDEX idx_events_created ON events(created_at DESC);

-- 또는 복합
CREATE INDEX idx_events_user_created
ON events(user_id, created_at DESC);
```

### 패턴 3: Soft Delete 대응

```sql
-- deleted_at이 있는 테이블
CREATE TABLE orders(
  id BIGINT PRIMARY KEY,
  user_id BIGINT,
  created_at TIMESTAMP,
  deleted_at TIMESTAMP
);

-- 활성 주문만 빠르게 검색
CREATE INDEX idx_orders_active_user
ON orders(user_id)
WHERE deleted_at IS NULL;
```

### 패턴 4: 상태 필터

```sql
-- status = 'active' 인 행이 대부분
CREATE TABLE subscriptions(
  id BIGINT PRIMARY KEY,
  user_id BIGINT,
  status TEXT,
  created_at TIMESTAMP
);

-- 활성 구독만 인덱싱
CREATE INDEX idx_subscriptions_user_active
ON subscriptions(user_id)
WHERE status = 'active';
```

### 패턴 5: 검색 (LIKE, 전문검색)

```sql
-- 이메일 검색
CREATE INDEX idx_users_email_text
ON users(email);

-- 테스트로 시작하는 이름 검색
SELECT * FROM users WHERE name LIKE 'test%';

-- 더 나은 방법: 전문검색 (GIN 인덱스)
CREATE INDEX idx_users_name_fts
ON users USING GIN(to_tsvector('english', name));

SELECT * FROM users
WHERE to_tsvector('english', name) @@ to_tsquery('english', 'test');
```

## CONCURRENTLY 옵션: 운영 중 안전한 인덱스 생성

### 일반 인덱스 생성의 문제

```sql
-- 이 작업 중 테이블이 LOCK됨
CREATE INDEX idx_large_table ON huge_table(column);
-- 위 쿼리가 끝날 때까지 INSERT/UPDATE/DELETE 불가능
-- → 운영 중 치명적
```

### CONCURRENTLY로 해결

```sql
-- 테이블에 LOCK을 잠깐만 걸고 나머지는 동시 진행
CREATE INDEX CONCURRENTLY idx_large_table
ON huge_table(column);

-- 이제 CREATE INDEX가 진행되는 동안에도
-- INSERT/UPDATE/DELETE 정상 작동
```

### CONCURRENTLY의 제한

```sql
-- ✅ CONCURRENTLY 사용 가능
CREATE INDEX CONCURRENTLY idx_name ON table(col);
DROP INDEX CONCURRENTLY idx_name;

-- ❌ CONCURRENTLY 사용 불가
CREATE UNIQUE INDEX CONCURRENTLY idx_name ON table(col);
-- (UNIQUE 인덱스는 일반적으로 불가능하지만 경우에 따라 가능)
```

### 실전 예제

```sql
-- 운영 중 새 인덱스 추가 (권장)
CREATE INDEX CONCURRENTLY idx_posts_created
ON posts(created_at DESC);

-- 운영 중 인덱스 제거 (권장)
DROP INDEX CONCURRENTLY idx_posts_old_index;
```

**규칙**: 프로덕션에서는 항상 `CONCURRENTLY`를 사용하세요.

## 핵심 정리

### 인덱스는...

| 개념 | 설명 |
|------|------|
| **목적** | 데이터를 빠르게 찾기 (Seq Scan → Index Scan) |
| **구조** | B-tree (정렬된 트리, 이진 검색 활용) |
| **생성** | `CREATE INDEX idx_name ON table(column);` |
| **삭제** | `DROP INDEX IF EXISTS idx_name;` |
| **비용** | 쓰기 느려짐, 디스크 증가, 메모리 사용 |

### 인덱스를 만들어야 할 때
- ✅ SELECT WHERE 조건에서 자주 사용
- ✅ JOIN 조건
- ✅ ORDER BY 정렬
- ✅ 범위 검색
- ✅ 카디널리티 높은 컬럼 (다양한 값)

### 인덱스를 피해야 할 때
- ❌ 카디널리티 매우 낮은 컬럼 (boolean, 성별)
- ❌ 자주 업데이트되는 컬럼
- ❌ 거의 쿼리되지 않는 컬럼
- ❌ 작은 테이블 (<1000행)

### Supabase 실전
- `user_id` + `created_at` 복합 인덱스가 가장 자주 사용됨
- Soft delete 테이블은 부분 인덱스로 최적화
- 운영 중 인덱스 작업은 `CONCURRENTLY` 필수
