# 05-A-04 유저 & 역할(Role) 생성/관리

> 유형: 실습 | 예상 소요 시간: 15~20분
> 이전: psql 핵심 명령어 모음 | 다음: GRANT / REVOKE 권한 부여와 회수 (그룹 B)

---

## 이 파일에서 하는 것

PostgreSQL에서 "유저 = 역할(Role)"이라는 개념을 이해하고, Supabase 셀프호스팅 환경에서 기본 역할들을 확인한 후 새로운 역할을 직접 만들고 관리해봅니다.

---

## 사전 조건

- Supabase 셀프호스팅이 실행 중 (`docker-compose up`)
- psql 클라이언트로 PostgreSQL에 접속할 수 있는 상태
- postgres 사용자 또는 충분한 권한을 가진 역할로 접속

---

## 실습

### Step 1. 현재 역할(Role) 확인하기

```bash
psql -U postgres -h localhost -d postgres
```

PostgreSQL 프롬프트가 나타나면:

```sql
\du
```

> `\du`는 psql의 메타 명령어로 현재 데이터베이스에 존재하는 모든 역할(Role)을 표시합니다.
>
> **출력 예시:**
> ```
>                    List of roles
>  Role name |                         Attributes
> -----------+---------------------------------------------------------
>  anon      | No inheritance, Cannot login
>  authenticated | No inheritance, Cannot login
>  postgres  | Superuser, Can login
>  service_role | Superuser, Can login
> ```
>
> **각 역할의 의미 (Supabase 기본 설정):**
> - **postgres**: 슈퍼유저. 모든 작업 가능
> - **anon**: 인증되지 않은 사용자 권한. 로그인 불가
> - **authenticated**: 인증된 사용자 권한. 로그인 불가
> - **service_role**: 백엔드 서비스용 슈퍼유저. 로그인 불가

---

### Step 2. PostgreSQL에서의 "유저 = 역할" 개념

npm에서 read-only와 admin 권한이 다르듯이, PostgreSQL에서도 각 역할마다 다른 권한을 가질 수 있습니다.

```sql
-- 현재 접속한 역할 확인
SELECT current_user;
```

> `current_user`는 현재 접속 중인 역할의 이름을 반환합니다.
> postgres 사용자로 접속했다면 `postgres`가 나타날 것입니다.

---

### Step 3. 새로운 역할(Role) 만들기

이제 테스트용으로 `app_user`라는 읽기 전용 역할을 만들어봅시다.

```sql
CREATE ROLE app_user;
```

> `CREATE ROLE` 명령어는 새로운 역할을 생성합니다.
> 이 명령어만으로는 로그인 권한을 주지 않습니다.

생성 후 다시 \du로 확인:

```sql
\du
```

> 새로운 `app_user` 역할이 목록에 추가되었을 것입니다.
> 속성 칼럼이 비어있다는 것은 특별한 권한이 없다는 뜻입니다.

---

### Step 4. 로그인 가능한 유저(User) 만들기

실제로 데이터베이스에 접속할 유저를 만들어봅시다.

```sql
CREATE USER dev_user WITH PASSWORD 'test123';
```

> `CREATE USER`는 `CREATE ROLE ... WITH LOGIN`의 축약형입니다.
> `WITH PASSWORD`로 비밀번호를 설정합니다.

확인:

```sql
\du
```

> `dev_user`가 "Can login" 속성과 함께 나타나야 합니다.

---

### Step 5. 비밀번호 변경하기

기존 역할의 비밀번호를 변경할 때:

```sql
ALTER ROLE dev_user WITH PASSWORD 'newpassword123';
```

> 변경하려는 역할이 이미 존재해야 합니다.

---

### Step 6. 역할의 속성 수정하기

`app_user` 역할에 로그인 권한을 추가:

```sql
ALTER ROLE app_user WITH LOGIN;
```

다시 확인:

```sql
\du
```

> `app_user`의 속성에 "Can login"이 추가되어야 합니다.

로그인 권한 제거:

```sql
ALTER ROLE app_user WITH NOLOGIN;
```

---

### Step 7. 역할의 상속 권한 설정

역할 간의 권한 상속을 관리할 수 있습니다.

```sql
-- 상속 불가 설정
ALTER ROLE app_user NOINHERIT;

-- 상속 가능 설정
ALTER ROLE app_user INHERIT;
```

> `INHERIT` / `NOINHERIT`는 이 역할이 다른 역할의 권한을 상속받을지 여부를 결정합니다.

---

### Step 8. 역할 삭제하기

더 이상 필요 없는 역할은 삭제할 수 있습니다.

```sql
DROP ROLE app_user;
```

> **주의:** 역할이 소유한 데이터베이스 객체(테이블, 스키마 등)가 있으면 삭제할 수 없습니다.
> 이 경우 먼저 그 객체들을 삭제하거나 다른 역할에 이관해야 합니다.

---

### Step 9. 생성한 역할로 접속 테스트

이전에 만든 `dev_user`로 실제 로그인을 시도해봅시다.

```bash
# 현재 postgres 접속을 종료
\q
```

새 터미널에서:

```bash
psql -U dev_user -h localhost -d postgres
```

> 비밀번호를 입력하라는 메시지가 나타나면, Step 4에서 설정한 `test123`을 입력합니다.

접속 성공 후:

```sql
SELECT current_user;
```

> `dev_user`가 반환되어야 합니다.

다시 postgres로 돌아가기:

```bash
\q
psql -U postgres -h localhost -d postgres
```

---

## 확인 방법

실습 완료 확인 체크리스트:

1. ✅ `\du` 명령어로 현재 역할 목록을 확인했는가
2. ✅ 새로운 역할 `app_user`를 생성했는가
3. ✅ 로그인 가능한 유저 `dev_user`를 생성했는가
4. ✅ `dev_user`의 비밀번호를 변경했는가
5. ✅ `app_user`의 속성을 수정했는가
6. ✅ `dev_user`로 실제 로그인했는가

---

## 자주 겪는 오류

| 오류 메시지 | 원인 | 해결 |
|-----------|------|------|
| `ERROR: role "..." already exists` | 이미 같은 이름의 역할이 존재 | 다른 이름으로 역할 생성하거나, 기존 역할 확인 |
| `ERROR: permission denied to create role` | 현재 역할에 역할 생성 권한이 없음 | postgres 또는 CREATEROLE 권한을 가진 역할로 접속 |
| `ERROR: role "..." cannot be dropped because some objects depend on it` | 삭제하려는 역할이 소유한 객체 존재 | `DROP OWNED BY app_user;` 실행 후 삭제 |
| `psql: error: FATAL: password authentication failed for user "dev_user"` | 비밀번호가 틀렸거나 존재하지 않는 유저 | 비밀번호 다시 확인하거나 `\du`로 역할 존재 여부 확인 |
| `ERROR: syntax error at or near "WITH"` | SQL 문법 오류 | 명령어를 정확히 확인 (예: `CREATE USER` vs `CREATE ROLE`) |
