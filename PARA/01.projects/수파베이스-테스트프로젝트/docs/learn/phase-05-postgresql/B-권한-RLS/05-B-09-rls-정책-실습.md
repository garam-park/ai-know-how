# 05-B-09 RLS 정책 작성 실습 (SELECT / INSERT 분리)

**유형**: 실습
**이전 과정**: Row Level Security (RLS) 개념
**다음 과정**: pg_dump로 수동 백업하기 (그룹 C)

---

## 학습 목표

- 실제 테이블에서 RLS(Row Level Security)를 활성화하는 방법 학습
- SELECT와 INSERT 정책을 분리하여 작성하기
- `auth.uid()` 함수를 활용하여 사용자별 데이터 제어
- 인증되지 않은(anon) 역할과 인증된(authenticated) 역할별 권한 테스트

---

## 실습 개요

이 실습에서는 `todos` 테이블을 생성하고, RLS 정책을 단계별로 적용하여 다음을 구현합니다:
- **SELECT 정책**: 사용자는 본인이 생성한 항목만 조회
- **INSERT 정책**: 사용자는 본인 user_id를 가진 행만 삽입 가능

### 개념 요약: RLS란?

RLS는 데이터베이스 수준에서 행(row) 단위의 접근 제어를 자동으로 처리합니다.
**FE 개발자 비유**: 일반적으로 서버 API에서 `WHERE user_id = req.user.id` 필터를 수동으로 추가하는 것처럼, RLS는 **데이터베이스가 자동으로 이 필터를 적용**해줍니다. FE에서 쿼리할 때 자신도 모르게 자신의 데이터만 조회되는 것입니다.

## 실습 단계

### 1단계: 테스트 테이블 생성

Supabase 대시보드에서 SQL 편집기를 열고 다음 쿼리를 실행합니다.

```sql
CREATE TABLE todos (
  id BIGSERIAL PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  description TEXT,
  is_completed BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 인덱스 생성 (성능 최적화)
CREATE INDEX idx_todos_user_id ON todos(user_id);
```

**완료 기준**:
- `todos` 테이블이 생성됨
- Supabase 대시보시의 "Table Editor"에서 `todos` 테이블이 보임

---

### 2단계: RLS 활성화

`todos` 테이블에 대해 RLS를 활성화합니다.

```sql
ALTER TABLE todos ENABLE ROW LEVEL SECURITY;
```

**확인 방법**:
```sql
-- RLS 활성화 상태 확인
SELECT
  tablename,
  rowsecurity
FROM pg_tables
WHERE tablename = 'todos';
```

**완료 기준**:
- 쿼리 결과에서 `rowsecurity = true`로 표시됨

**⚠️ 중요**: RLS를 활성화하면 정책이 없을 경우 모든 데이터 접근이 차단됩니다!

---

### 3단계: SELECT 정책 작성

사용자는 자신이 생성한 항목만 조회할 수 있는 정책을 만듭니다.

```sql
CREATE POLICY select_own_todos ON todos
  FOR SELECT
  USING (auth.uid() = user_id);
```

**정책 설명**:
- **정책 이름**: `select_own_todos`
- **대상 작업**: SELECT (읽기)
- **조건**: `auth.uid() = user_id` → 현재 로그인한 사용자의 UID가 행의 `user_id`와 일치할 때만 조회 가능

**확인 방법**:
```sql
-- 정책 확인
SELECT * FROM pg_policies WHERE tablename = 'todos';
```

---

### 4단계: INSERT 정책 작성

사용자는 자신의 user_id를 가진 행만 삽입할 수 있는 정책을 만듭니다.

```sql
CREATE POLICY insert_own_todos ON todos
  FOR INSERT
  WITH CHECK (auth.uid() = user_id);
```

**정책 설명**:
- **정책 이름**: `insert_own_todos`
- **대상 작업**: INSERT (삽입)
- **조건**: `WITH CHECK (auth.uid() = user_id)` → 삽입 시 `user_id` 값이 현재 사용자 UID와 일치해야 함

---

### 5단계: 정책 테스트 (인증 사용자)

Supabase 프로젝트의 API 문서나 클라이언트 라이브러리를 사용하여 정책을 테스트합니다.

#### 5-1. 테스트 데이터 삽입 (관리자 권한)

먼저 테스트용 데이터를 삽입합니다. 이를 위해 RLS 정책을 **임시로 비활성화**하거나 관리자 권한으로 실행합니다.

```sql
-- 임시로 RLS 비활성화 (테스트 데이터 삽입용)
ALTER TABLE todos DISABLE ROW LEVEL SECURITY;

-- 테스트 데이터 삽입
-- 실제 user_id는 auth.users 테이블의 실제 ID로 변경하세요
INSERT INTO todos (user_id, title, description, is_completed)
VALUES
  ('11111111-1111-1111-1111-111111111111', 'User A의 할 일 1', 'Description 1', FALSE),
  ('11111111-1111-1111-1111-111111111111', 'User A의 할 일 2', 'Description 2', TRUE),
  ('22222222-2222-2222-2222-222222222222', 'User B의 할 일 1', 'Description 3', FALSE);

-- RLS 다시 활성화
ALTER TABLE todos ENABLE ROW LEVEL SECURITY;
```

#### 5-2. JavaScript/TypeScript 클라이언트 테스트

```typescript
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  'YOUR_SUPABASE_URL',
  'YOUR_ANON_KEY'
);

// User A로 로그인했다고 가정
await supabase.auth.signInWithPassword({
  email: 'usera@example.com',
  password: 'password123'
});

// SELECT 정책 테스트: User A는 자신의 데이터만 조회
const { data, error } = await supabase
  .from('todos')
  .select('*');

console.log('조회 결과:', data);
// 예상: User A의 2개 항목만 반환 (User B의 항목은 포함되지 않음)
```

#### 5-3. INSERT 정책 테스트 (성공 케이스)

```typescript
// 자신의 user_id로 행 삽입 (성공)
const { data, error } = await supabase
  .from('todos')
  .insert([
    {
      user_id: supabase.auth.user()?.id,  // 현재 사용자의 ID
      title: 'My new todo',
      description: 'This is my todo'
    }
  ]);

console.log('삽입 성공:', data);
```

#### 5-4. INSERT 정책 테스트 (실패 케이스)

```typescript
// 다른 사용자의 user_id로 행 삽입 시도 (실패)
const { data, error } = await supabase
  .from('todos')
  .insert([
    {
      user_id: '22222222-2222-2222-2222-222222222222',  // 다른 사용자의 ID
      title: 'Hack attempt',
      description: 'Try to insert as someone else'
    }
  ]);

console.log('에러 발생:', error?.message);
// 예상: "new row violates row level security policy"
```

**테스트 완료 기준**:
- ✅ 자신의 데이터만 SELECT 가능
- ✅ 자신의 user_id로 INSERT 가능
- ✅ 다른 사용자의 user_id로 INSERT 불가능 (에러 발생)

### 6단계: 익명(anon) 역할 정책 확인

인증되지 않은 사용자의 접근을 테스트합니다.

```typescript
// 로그아웃
await supabase.auth.signOut();

// 인증 없이 조회 시도
const { data, error } = await supabase
  .from('todos')
  .select('*');

console.log('에러:', error?.message);
// 예상: "permission denied" 또는 no rows returned
```

**예상 동작**:
- 현재 정책에는 익명 사용자를 위한 정책이 없으므로 접근 불가능

---

### 7단계: 정책 수정

기존 정책을 변경하려면 먼저 삭제 후 다시 생성합니다.

#### 7-1. 정책 삭제

```sql
DROP POLICY select_own_todos ON todos;
DROP POLICY insert_own_todos ON todos;
```

#### 7-2. 새 정책 생성 (예: UPDATE 정책 추가)

```sql
CREATE POLICY update_own_todos ON todos
  FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY delete_own_todos ON todos
  FOR DELETE
  USING (auth.uid() = user_id);
```

### 8단계: 정책 조회 및 관리

생성된 모든 정책을 확인합니다.

```sql
-- todos 테이블의 모든 정책 조회
SELECT
  policyname,
  permissive,
  cmd,
  qual,
  with_check
FROM pg_policies
WHERE tablename = 'todos'
ORDER BY policyname;
```

**출력 예시**:
```
      policyname       | permissive |  cmd   |        qual        |     with_check
---------------------+------------+--------+--------------------+--------------------

 insert_own_todos     | PERMISSIVE | INSERT | (null)             | (auth.uid() =...
 select_own_todos     | PERMISSIVE | SELECT | (auth.uid() =...   | (null)
 update_own_todos     | PERMISSIVE | UPDATE | (auth.uid() =...   | (auth.uid() =...
```

## 정책 삭제하기

학습이 끝나면 테스트 정책과 테이블을 정리합니다.

```sql
-- 모든 정책 삭제
DROP POLICY IF EXISTS select_own_todos ON todos;
DROP POLICY IF EXISTS insert_own_todos ON todos;
DROP POLICY IF EXISTS update_own_todos ON todos;
DROP POLICY IF EXISTS delete_own_todos ON todos;

-- 테이블 삭제 (선택사항)
DROP TABLE IF EXISTS todos CASCADE;
```

## 핵심 개념 정리

| 개념 | 설명 |
|------|------|
| **RLS** | 데이터베이스가 사용자별로 자동으로 행 필터링 |
| **auth.uid()** | 현재 로그인한 사용자의 UUID 반환 |
| **USING 절** | SELECT/DELETE 및 UPDATE의 조회 조건 |
| **WITH CHECK** | INSERT/UPDATE 시 데이터 삽입 전 검증 |
| **정책 없음** | RLS 활성화 후 정책이 없으면 모든 접근 차단 |

## 자주하는 질문 (FAQ)

**Q1: RLS 정책이 없으면 어떻게 되나요?**
A: RLS를 활성화한 후 정책이 하나도 없으면, 관리자(postgres 역할)를 제외한 모든 사용자는 해당 테이블에 접근할 수 없습니다. 반드시 정책을 만들어야 합니다.

**Q2: auth.uid()가 NULL이 되는 경우는?**
A: 인증되지 않은(anon) 사용자일 때 `auth.uid()`는 NULL입니다. 익명 사용자 허용 정책이 필요하면 `auth.uid() IS NULL` 조건을 사용합니다.

**Q3: 정책 성능은 어떻게 되나요?**
A: RLS 정책은 데이터베이스 수준에서 처리되므로, 어플리케이션 코드에서 필터링하는 것보다 효율적입니다. 단, 복잡한 정책은 성능 영향이 있을 수 있으므로 인덱싱(예: `idx_todos_user_id`)이 중요합니다.

**Q4: 관리자 계정에는 RLS가 적용되나요?**
A: 기본적으로 `postgres` 역할(DB 소유자)은 RLS 정책의 영향을 받지 않습니다. 하지만 다른 역할(예: `authenticated`, `anon`)은 정책을 따릅니다.

## 관련 명령어 빠른 참고

```bash
# RLS 활성화
ALTER TABLE {table_name} ENABLE ROW LEVEL SECURITY;

# RLS 비활성화
ALTER TABLE {table_name} DISABLE ROW LEVEL SECURITY;

# 정책 생성 (SELECT)
CREATE POLICY {policy_name} ON {table_name}
  FOR SELECT
  USING ({condition});

# 정책 생성 (INSERT)

  FOR INSERT
  WITH CHECK ({condition});

# 정책 삭제
DROP POLICY {policy_name} ON {table_name};

# 모든 정책 조회
SELECT * FROM pg_policies WHERE tablename = '{table_name}';
```
