# 권한 통합테스트

## 예상 소요시간
| 구분 | 시간 |
|------|------|
| 사람 (숙련 개발자) | ~30분 |
| AI 에이전트 | ~10분 |

## What (무엇을)
Role/Permission/Scope 조합이 복잡하므로, 전체 권한 시스템의 통합 흐름을 테스트합니다. 5개 시나리오로 권한 시스템이 정상 동작함을 검증합니다.

## Why (왜)
권한 시스템은 여러 계층(Role, Permission, Scope, ProjectRolePermission)이 복합적으로 동작하므로, 조기에 통합 테스트를 작성하여 버그를 예방하고 신뢰도를 높입니다.

## 선행 작업

## 수행 가이드

### Step 1: 통합 테스트 파일 생성
- 대상 파일: `apps/backend/src/modules/permission/permission.integration.spec.ts`
- 작업 내용: Jest + supertest를 이용한 통합 테스트 구조
  ```typescript
  import { INestApplication } from '@nestjs/common';
  import { Test, TestingModule } from '@nestjs/testing';
  import * as request from 'supertest';
  import { AppModule } from '../../app.module';
  import { PrismaService } from '../prisma/prisma.service';

  describe('Permission Integration Tests', () => {
    let app: INestApplication;
    let prisma: PrismaService;
    let projectId: string;
    let companyIds: string[] = [];
    let userIds: string[] = [];
    let accessTokens: Record<string, string> = {};

    beforeAll(async () => {
      const moduleFixture: TestingModule = await Test.createTestingModule({
        imports: [AppModule],
      }).compile();

      app = moduleFixture.createNestApplication();
      prisma = moduleFixture.get<PrismaService>(PrismaService);
      await app.init();
    });

    afterAll(async () => {
      // 테스트 데이터 정리
      await prisma.projectMember.deleteMany();
      await prisma.project.deleteMany();
      await prisma.user.deleteMany();
      await prisma.company.deleteMany();
      await app.close();
    });

    beforeEach(async () => {
      // 테스트용 회사 생성
      const company1 = await prisma.company.create({
        data: {
          name: '주관사',
          registrationNumber: '12345678',
          role: 'OWNER',
        },
      });

      const company2 = await prisma.company.create({
        data: {
          name: '참여사',
          registrationNumber: '87654321',
          role: 'MEMBER',
        },
      });

      companyIds = [company1.id, company2.id];

      // 테스트용 프로젝트 생성
      const project = await prisma.project.create({
        data: {
          name: '테스트 프로젝트',
          startDate: new Date('2024-01-01'),
          endDate: new Date('2024-12-31'),
          owner: { connect: { id: company1.id } },
          companies: {
            connect: [{ id: company1.id }, { id: company2.id }],
          },
        },
      });

      projectId = project.id;

      // 각 Role별 사용자 생성 및 로그인
      const roles = ['주관사PM', '참여사PM', '실무자', '뷰어'];
      for (let i = 0; i < roles.length; i++) {
        const email = `${roles[i].toLowerCase()}@test.com`;
        const user = await prisma.user.create({
          data: {
            email,
            password: 'hashed_password',
            name: roles[i],
            tel: '010-0000-0001',
          },
        });

        userIds.push(user.id);

        // ProjectMember 추가 (roleId = i + 1)
        await prisma.projectMember.create({
          data: {
            projectId,
            userId: user.id,
            roleId: i + 1,
          },
        });

        // 토큰 발급 (테스트에서 JWT 검증은 mock)
        accessTokens[roles[i]] = `token-for-${roles[i]}`;
      }
    });

    // 테스트 시나리오들이 이어집니다
  });
  ```

### Step 2: 시나리오 1 - 주관사PM 모든 권한 (ALL scope)
- 대상 파일: `apps/backend/src/modules/permission/permission.integration.spec.ts`
- 작업 내용: 주관사PM이 WBS를 조회하면 ALL scope 반환
  ```typescript
  describe('Scenario 1: 주관사PM - WBS VIEW → ALL scope', () => {
    it('주관사PM이 WBS 조회 시 ALL scope 반환', async () => {
      const response = await request(app.getHttpServer())
        .get(`/api/v1/projects/${projectId}/wbs`)
        .set('Authorization', `Bearer token-for-주관사PM`)
        .expect(200);

      expect(response.body.code).toBe(200000);
      expect(response.body.result.scope).toBe('ALL');
      expect(response.body.result.wbs).toBeDefined();
    });

    it('주관사PM이 WBS 편집 가능', async () => {
      const response = await request(app.getHttpServer())
        .patch(`/api/v1/projects/${projectId}/wbs/wbs-1`)
        .set('Authorization', `Bearer token-for-주관사PM`)
        .send({ title: '수정된 WBS' })
        .expect(200);

      expect(response.body.code).toBe(200000);
    });
  });
  ```

### Step 3: 시나리오 2 - 참여사PM 범위 제한 (OWN_COMPANY scope)
- 작업 내용: 참여사PM이 WBS를 조회하면 OWN_COMPANY scope 반환
  ```typescript
  describe('Scenario 2: 참여사PM - WBS → OWN_COMPANY scope', () => {
    it('참여사PM이 WBS 조회 시 OWN_COMPANY scope 반환', async () => {
      const response = await request(app.getHttpServer())
        .get(`/api/v1/projects/${projectId}/wbs`)
        .set('Authorization', `Bearer token-for-참여사PM`)
        .expect(200);

      expect(response.body.code).toBe(200000);
      expect(response.body.result.scope).toBe('OWN_COMPANY');
      // WBS 데이터는 자사만 포함되어야 함 (컨트롤러에서 필터링)
    });

    it('참여사PM이 타사 WBS 편집 시도하면 실패', async () => {
      // 타사 WBS를 생성하고 편집 시도
      const response = await request(app.getHttpServer())
        .patch(`/api/v1/projects/${projectId}/wbs/other-company-wbs`)
        .set('Authorization', `Bearer token-for-참여사PM`)
        .send({ title: '타사 WBS 편집' })
        .expect(403); // 컨트롤러에서 scope 기반 재검증

      expect(response.body.code).toBe(403002);
    });
  });
  ```

### Step 4: 시나리오 3 - 실무자 할당 작업만 (OWN_TASK scope)
- 작업 내용: 실무자가 WBS를 조회하면 OWN_TASK scope 반환
  ```typescript
  describe('Scenario 3: 실무자 - WBS → OWN_TASK scope', () => {
    it('실무자가 WBS 조회 시 OWN_TASK scope 반환', async () => {

        .set('Authorization', `Bearer token-for-실무자`)

      expect(response.body.result.scope).toBe('OWN_TASK');
      // WBS 데이터는 할당된 작업만 포함

    it('실무자가 할당되지 않은 작업 편집 시도하면 실패', async () => {

        .set('Authorization', `Bearer token-for-실무자`)
        .send({ title: '할당되지 않은 작업 편집' })
        .expect(403);

      expect(response.body.code).toBe(403002);
    });
  });
  ```

### Step 5: 시나리오 4 - 뷰어 권한 없음
- 작업 내용: 뷰어가 조회 이외 작업을 시도하면 권한 없음
  ```typescript
  describe('Scenario 4: 뷰어 - WBS 편집 불가', () => {
    it('뷰어가 WBS 편집 시도하면 403 에러', async () => {
      const response = await request(app.getHttpServer())
        .patch(`/api/v1/projects/${projectId}/wbs/wbs-1`)
        .set('Authorization', `Bearer token-for-뷰어`)
        .send({ title: '편집 시도' })
        .expect(403);

      expect(response.body.code).toBe(403001);
      expect(response.body.message).toContain('WBS EDIT');
    });

    it('뷰어가 WBS 조회는 가능 (ALL scope)', async () => {
      const response = await request(app.getHttpServer())
        .get(`/api/v1/projects/${projectId}/wbs`)
        .set('Authorization', `Bearer token-for-뷰어`)
        .expect(200);

      expect(response.body.code).toBe(200000);
      expect(response.body.result.scope).toBe('ALL');
    });

    it('뷰어가 보고서는 조회 가능', async () => {
      const response = await request(app.getHttpServer())
        .get(`/api/v1/projects/${projectId}/reports`)
        .set('Authorization', `Bearer token-for-뷰어`)
        .expect(200);

      expect(response.body.code).toBe(200000);
    });
  });
  ```

### Step 6: 시나리오 5 - ProjectRolePermission 덮어쓰기
- 작업 내용: 프로젝트 커스텀 권한이 기본값을 덮어쓰는지 확인
  ```typescript
  describe('Scenario 5: ProjectRolePermission 우선순위', () => {
    beforeEach(async () => {
      // 참여사PM의 WBS VIEW 권한을 OWN_TASK로 커스터마이징
      const permission = await prisma.permission.findUnique({
        where: {
          resource_action: {
            resource: 'WBS',
            action: 'VIEW',
          },
        },
      });

      await prisma.projectRolePermission.create({
        data: {
          projectId,
          roleId: 2, // 참여사PM
          permissionId: permission.id,
          scope: 'OWN_TASK',
        },
      });
    });

    it('ProjectRolePermission이 RolePermission을 덮어씀', async () => {
      const response = await request(app.getHttpServer())
        .get(`/api/v1/projects/${projectId}/wbs`)
        .set('Authorization', `Bearer token-for-참여사PM`)
        .expect(200);

      // 기본값 OWN_COMPANY에서 커스텀 OWN_TASK로 변경됨
      expect(response.body.result.scope).toBe('OWN_TASK');
    });
  });
  ```

## 참고
- 통합 테스트는 실제 데이터베이스를 사용하므로 테스트 DB 분리 권장 (test.env 사용).
- beforeEach에서 테스트용 데이터를 매번 생성하여 테스트 독립성 보장.
- afterAll에서 테스트 데이터를 정리하여 다른 테스트에 영향 최소화.
- 실제 JWT 검증이 필요하면 TestingModule에서 JwtService를 mock하거나 실제 토큰 발급.
- 성능 최적화: 테스트가 느리면 데이터베이스 Seed 최적화나 테스트 병렬 실행 검토.
