# 운영환경 배포

## 예상 소요시간
| 구분 | 시간 |
|------|------|
| 사람 (숙련 개발자) | ~40분 |
| AI 에이전트 | ~15분 |

## What (무엇을)
클라우드 또는 온프레미스 운영 환경에 애플리케이션을 배포합니다. 환경 변수 관리, DB 마이그레이션, SSL/도메인 설정, 모니터링 기본 설정을 포함합니다.

## Why (왜)
실제 사용자에게 안정적으로 서비스를 제공하기 위한 최종 단계입니다.

## 수행 가이드

### Step 1: 배포 대상 검토
운영 환경 선택지:
- AWS (EC2, ECS, EKS)
- Google Cloud (Compute Engine, GKE)
- Azure (App Service, AKS)
- Naver Cloud Platform (NCP)
- 온프레미스 (VPS, 자체 서버)

이 가이드는 기본적인 Linux VPS 기준으로 작성합니다.

### Step 2: 서버 초기 설정
SSH로 서버 접속 후:

```bash
# 시스템 업데이트
sudo apt update && sudo apt upgrade -y

# Docker 설치
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh

# Docker Compose 설치
sudo curl -L "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)" -o /usr/local/bin/docker-compose
sudo chmod +x /usr/local/bin/docker-compose

# 사용자 권한 설정
sudo usermod -aG docker $USER

# 재부팅 (권한 적용)
sudo reboot

# Certbot (SSL) 설치
sudo apt install certbot python3-certbot-nginx -y

# Git 설치
sudo apt install git -y

# Node.js 설치 (선택)
curl -sL https://deb.nodesource.com/setup_18.x | sudo -E bash -
sudo apt install -y nodejs
```

### Step 3: 애플리케이션 배포
```bash
# 배포 디렉토리 생성
mkdir -p /srv/app
cd /srv/app

# 저장소 클론
git clone <repository-url> .

# 운영 환경 변수 파일 생성
cat > .env.production << 'EOF'
NODE_ENV=production
DATABASE_URL=postgresql://prod_user:${DB_PASSWORD}@postgres:5432/prod_db
JWT_SECRET=${JWT_SECRET}
JWT_EXPIRY=3600
REFRESH_TOKEN_EXPIRY=604800
CORS_ORIGIN=https://example.com
PORT=3001
TZ=Asia/Seoul
EOF

# Docker Compose 운영 파일 생성
cat > docker-compose.prod.yml << 'EOF'
version: '3.8'

services:
  postgres:
    image: postgres:15-alpine
    container_name: prod-postgres
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
      TZ: Asia/Seoul
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./backups:/backups
    restart: always
    networks:
      - app_network
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER}"]
      interval: 10s
      timeout: 5s
      retries: 5

  backend:
    image: ${DOCKER_USERNAME}/app-backend:${VERSION}
    container_name: prod-backend
    environment:
      NODE_ENV: production
      DATABASE_URL: ${DATABASE_URL}
      JWT_SECRET: ${JWT_SECRET}
      PORT: 3001
    restart: always
    depends_on:
      postgres:
        condition: service_healthy
    networks:
      - app_network
    healthcheck:
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "--spider", "http://localhost:3001/health"]
      interval: 30s
      timeout: 5s
      retries: 3

  nginx:
    image: nginx:alpine
    container_name: prod-nginx
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.prod.conf:/etc/nginx/conf.d/default.conf
      - ./ssl:/etc/nginx/ssl
      - nginx_cache:/var/cache/nginx
    restart: always
    depends_on:
      - backend
    networks:
      - app_network

volumes:
  postgres_data:
  nginx_cache:

networks:
  app_network:
```

### Step 4: Nginx 운영 설정
`nginx.prod.conf` 파일을 생성합니다:

```nginx
upstream backend {
    server backend:3001;
}

# HTTP to HTTPS 리다이렉트
server {
    listen 80;
    server_name example.com www.example.com;

    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

# HTTPS
server {
    listen 443 ssl http2;
    server_name example.com www.example.com;

    # SSL 인증서
    ssl_certificate /etc/nginx/ssl/fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;

    # SSL 보안 설정
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    # HSTS
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # 보안 헤더
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;

    # 로깅
    access_log /var/log/nginx/access.log;
    error_log /var/log/nginx/error.log;

    # 속도 제한
    limit_req_zone $binary_remote_addr zone=general:10m rate=10r/s;
    limit_req_zone $binary_remote_addr zone=api:10m rate=30r/s;
    limit_req zone=general burst=20 nodelay;

    # 캐싱
    proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=api_cache:10m max_size=1g inactive=60m;

    # API 프록시
    location /api/ {
        limit_req zone=api burst=50 nodelay;

        proxy_pass http://backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 타임아웃 설정
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;

        # 캐싱 (GET 요청만)
        proxy_cache api_cache;
        proxy_cache_valid 200 5m;
        proxy_cache_use_stale error timeout invalid_header updating http_500 http_502 http_503 http_504;
    }

    # 정적 파일
    location / {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /index.html;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # 헬스 체크 엔드포인트
    location /health {
        access_log off;
        return 200 "healthy\n";
        add_header Content-Type text/plain;
    }
}
```

### Step 5: SSL 인증서 설정
```bash
# Let's Encrypt 인증서 발급
sudo certbot certonly --webroot -w /srv/app \
  -d example.com -d www.example.com \
  --email admin@example.com \
  --agree-tos --non-interactive

# 인증서 복사
sudo cp /etc/letsencrypt/live/example.com/fullchain.pem /srv/app/ssl/
sudo cp /etc/letsencrypt/live/example.com/privkey.pem /srv/app/ssl/
sudo chown -R 1000:1000 /srv/app/ssl/

# 자동 갱신 설정 (crontab)
# 매월 1일 자동 갱신
0 2 1 * * certbot renew --quiet
```

### Step 7: 데이터베이스 백업 및 복구
```bash
# 자동 백업 스크립트 생성
cat > /srv/app/backup.sh << 'EOF'
#!/bin/bash

BACKUP_DIR="/srv/app/backups"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="$BACKUP_DIR/backup_$TIMESTAMP.sql.gz"

# 백업 디렉토리 생성
mkdir -p $BACKUP_DIR

# 백업 수행
docker-compose -f docker-compose.prod.yml exec -T postgres \
  pg_dump -U prod_user prod_db | gzip > $BACKUP_FILE

# 오래된 백업 삭제 (30일 이상)
find $BACKUP_DIR -name "backup_*.sql.gz" -mtime +30 -delete

echo "Backup completed: $BACKUP_FILE"
EOF

chmod +x /srv/app/backup.sh

# Crontab에 추가 (매일 02:00 백업)
# 0 2 * * * /srv/app/backup.sh
```

### Step 8: 배포 스크립트
`deploy.sh` 파일을 생성합니다:

```bash
#!/bin/bash

set -e

VERSION=$1

if [ -z "$VERSION" ]; then
    echo "Usage: ./deploy.sh <version>"
    exit 1
fi

echo "Deploying version: $VERSION"

# 환경 변수 로드
set -a
source .env.production
source .env.secrets
set +a

# 현재 버전 백업
if [ -d "current" ]; then
    mv current previous_$(date +%s)
fi

# 새 버전 다운로드
git fetch origin
git checkout v$VERSION

# 환경 변수 설정
export VERSION=$VERSION

# Docker 이미지 풀
docker pull ${DOCKER_USERNAME}/app-backend:${VERSION}
docker pull ${DOCKER_USERNAME}/app-frontend:${VERSION}

# 헬스 체크 함수
health_check() {
    for i in {1..30}; do
        if curl -f http://localhost:3001/health > /dev/null 2>&1; then
            echo "✓ Backend is healthy"
            return 0
        fi
        echo "Waiting for backend... ($i/30)"
        sleep 2
    done
    echo "✗ Backend health check failed"
    return 1
}

# 배포
echo "Stopping old containers..."
docker-compose -f docker-compose.prod.yml down || true

echo "Starting new containers..."
docker-compose -f docker-compose.prod.yml up -d

echo "Running migrations..."
sleep 5  # DB 시작 대기
docker-compose -f docker-compose.prod.yml exec -T backend npx prisma migrate deploy

# 헬스 체크
if health_check; then
    echo "✓ Deployment successful!"
    exit 0
else
    echo "✗ Deployment failed, rolling back..."
    ./rollback.sh
    exit 1
fi
```
