# 공공 IT 발주 웹서비스 - 테스트 (Test) 가이드

본 문서는 프로젝트의 안정성을 보장하기 위해 도입된 **자동화 E2E (End-to-End) 테스트**의 목적(Why), 대상(What), 그리고 실행 방법(How)에 대해 안내합니다.

---

## 🤔 Why: 왜 자동화 테스트를 작성해야 하나요?

1. **상호작용 검증 (UI/UX 무결성 보장)** 
   - 프론트엔드와 백엔드가 결합된 최종 사용자 관점에서 플로우(회사 등록 → 프로젝트 생성 → 컨소시엄 추가 → 멤버 초대 등)가 올바르게 작동하는지 보장합니다.
2. **회귀 버그 (Regression Bug) 방지**
   - 새로운 기능이 추가되거나 기존 코드가 리팩터링될 때, 알 수 없는 사이드 이펙트로 인해 기존 핵심 기능이 망가졌는지(회귀) 즉각적으로 파악할 수 있게 해줍니다. 
3. **멱등성(Idempotency)의 확보**
   - 수동 테스트는 사람이 직접 DB를 지우고 새로 계정을 만들어야 하는 반복적인 비용이 듭니다. 현재 구축된 E2E 셋업은 **도커 컨테이너 격리 및 수행 전 DB 자동 초기화** 스크립트가 내장되어 있어 "외부 요인 없이 언제든 동일한 결과를 보장"합니다.

---

## 🎯 What: 무엇을 테스트하나요?

모든 UI를 일일이 테스트 코드로 작성하면 비용 효율성이 떨어집니다. 따라서 본 프로젝트는 **핵심 비즈니스 해피 패스(Happy Path)**를 중점적으로 E2E 코드화(`Playwright` 사용) 했습니다.

현재 `apps/frontend/e2e` 경로에 아래 3개의 핵심 시나리오가 묶여 있습니다:
1. **`01-company-and-project.spec.ts`**: 회사 등록 및 생성된 회사를 주관사로 지정한 프로젝트 생성 프로세스 통과.
2. **`02-consortium-and-member.spec.ts`**: 컨소시엄사 트리 구성 및 역할군을 지닌 멤버의 초대 기능 상호작용 검증.
3. **`03-wbs-management.spec.ts`**: 프로젝트 내 WBS 작업의 생성, 자식 노드 확장, 진척율 숫자 편집 시 계산 로직 검증.

---

## 🚀 How: 어떻게 테스트를 실행하나요?

이 프로젝트는 현재 로컬 환경의 오염을 방지하기 위해 **테스트 전용 컨테이너(`docker-compose.test.yml`) 세트**를 띄우고 테스트를 수행합니다. 
진행은 모두 자동으로 이뤄지니 아래 명령어만 기억하시면 됩니다.

### 시작하기 앞서 (Prerequisites)
1. Docker Desktop(또는 Docker daemon)이 실행 중이어야 합니다.
2. 현재 터미널이 **`apps/frontend/`** 폴더 안에 위치해 있어야 합니다.

### 실행 명령어

| 명령어 | 설명 | 활용 시기 |
|--------|------|-----------|
| `npm run test:e2e` | **헤드리스(Headless) 백그라운드 테스트 수행**. <br>도커 컨테이너를 스핀업하고, 뒷단에서 3가지 시나리오를 고속 병렬로 수행한 뒤, 결과를 출력하고 도커를 자동 종료합니다. | 평소 코드 수정/리팩토링 결과를 터미널에서 빠르게 검증할 때. 혹은 CI서버 런타임 용. |
| `npm run test:e2e:ui` | **브라우저 UI 시각적 테스트 수행**. <br>Playwright의 시각화 툴이 열려 마치 사람이 직접 브라우저를 켜고 광클릭하듯 각 DOM 요소의 움직임을 두 눈으로 추적·디버그할 수 있습니다. | E2E 테스트 코드 자체가 터질 때. 클릭하는 위치가 보이지 않아 상세 디버깅이 필요할 때. |
| `npm run test:e2e:up` | **테스트 환경 도커만 수동 시작**. | 개발 중 Playwright Inspector만 따로 돌리고 싶을 때 활용. (포트는 3003을 띄웁니다) |
| `npm run test:e2e:down`| **테스트 도커 수동 종료 및 볼륨 삭제**. | 중간에 비정상 종료되었거나 테스트 DB(test-db)가 좀비 프로세스로 남아있을 때 활용. |

### 예시: 스크립트 실행 흐름의 내막
1. `npm run test:e2e` 입력 시 내부적으로 `global-setup.ts`가 동작합니다.
2. `public-it-test-backend` 컨테이너(포트 3002)와 DB(포트 5433)를 이용해 `npx prisma migrate reset` 및 초깃값 시딩(`seed.ts`)을 동작시킵니다.
3. 데이터가 깔끔해지면 `playwright`가 3개의 Spec 시나리오를 바탕으로 **http://localhost:3003** (테스트 플랫폼 프론트엔드) 에 접속해 화면을 자동 검수합니다.
4. 모든 검열을 통과하면 자동으로 `docker-compose.test.yml down`되어 찌꺼기를 정리합니다.
