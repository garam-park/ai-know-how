## 학습 목표

- Runbook이 무엇인지 이해하기
- FE 개발자가 운영 중 만날 만한 장애 대응하기
- 각 장애별 디버깅 절차 작성하기
- 팀과 공유 가능한 템플릿 만들기
- Post-mortem 문서 작성하기

## FE 개발자를 위한 유추

프론트엔드 개발 중 버그를 만나면 어떻게 하나요?

```javascript
// 문제: 페이지 로딩이 안 됨
// → DevTools > Console에서 에러 확인
// → Network 탭에서 API 응답 확인
// → 문제 원인 파악 후 해결
```

Runbook은 **팀 전체가 따를 수 있는 프로덕션 디버깅 가이드**입니다. 차이점은:

- **FE 디버깅**: 개인이 한 번에 해결 (또는 다른 개발자에게 물어봄)
- **Runbook**: 심야에 고장 났을 때, 팀원이 따라 하기만 해도 문제 해결

---

## 핵심 개념

### 1. Runbook이란?

**Runbook의 정의:**

```
일련의 자동화된 절차들을 담은 운영 가이드
→ IT 운영팀이 반복되는 작업이나 장애 대응 시 참고
→ 누구나 같은 방식으로 대응하게 하는 표준 운영 절차(SOP)
```

**왜 필요할까?**

| 상황 | Runbook 없을 때 | Runbook 있을 때 |
|------|-----------------|-----------------|
| DB 다운 | 혼란 스럽고, 복구 30분 | 명확한 절차, 복구 5분 |
| 야간 장애 | 개발자 깨우기, 불안감 | 누구든 대응 가능 |
| 신입 온보딩 | 직접 배우기, 오류 가능 | 문서 읽고 따라하기 |
| 팀 협업 | 각자 다른 방식 | 표준화된 방식 |

---

### 2. Runbook 구조

**최소 필수 요소:**

```markdown
# [장애 이름]

## 증상
- 사용자가 보는 증상
- 모니터링에서 보이는 신호

## 영향도
- 영향 범위 (전체/특정 사용자/특정 기능)
- 중요도 (Critical/High/Medium/Low)

## 원인 추론
- 일반적인 원인 3-5가지

## 진단 (Diagnosis)
- 단계별 확인 방법

## 해결 (Resolution)
- 해결 절차

## 검증 (Verification)
- 문제 해결 확인

## Escalation
- 직접 해결 안 될 때 누구에게 알릴 것

## Post-mortem
- 사건 후 분석 항목
```

### 3. 주요 장애 5가지와 Runbook

### **장애 1: PostgreSQL (Database) 다운**

```markdown
# [Runbook] PostgreSQL이 응답하지 않음

## 증상
- 앱에서 "Connection timeout" 에러
- /api/health 응답 없음
- 클라이언트: "Cannot connect to database"

## 영향도
- **범위**: 전체 서비스 (모든 사용자)
- **중요도**: CRITICAL (즉시 대응 필요)

## 원인 추론
1. PostgreSQL 프로세스 충돌
2. 디스크 공간 부족 (DB 쓰기 불가)
3. 메모리 부족 (OOM kill)
4. 네트워크 연결 끊김
5. 파일 권한 문제

## 진단

### Step 1: 컨테이너 상태 확인 (1분)
\`\`\`bash
docker ps | grep postgres

# 출력 1: 정상 (Up)
# supabase-postgres   Up 45 days

# 출력 2: 비정상 (Exited)
# supabase-postgres   Exited (1) 2 minutes ago
\`\`\`

**해석:**
- `Up` → 프로세스는 실행 중 (다음 Step 2로)
- `Exited` → 프로세스 충돌 (Step 3으로)

### Step 2: PostgreSQL 응답 확인 (1분)
\`\`\`bash
docker exec supabase-postgres pg_isready -U postgres

# 출력 1: 정상
# accepting connections

# 출력 2: 비정상
# rejecting connections
\`\`\`

**해석:**
- `accepting connections` → 프로세스는 살아있으나 포트 문제 가능 (Step 4로)
- `rejecting connections` → DB 자체 문제 (Step 3으로)

### Step 3: 로그 확인 (5분)
\`\`\`bash
docker logs supabase-postgres --tail=100 | grep -i error

# 주요 에러 메시지 찾기
# - "out of memory"
# - "no space left on device"
# - "permission denied"
\`\`\`

### Step 4: 리소스 확인 (2분)
\`\`\`bash
# 디스크 확인
df -h /
# 80% 이상 차있으면 디스크 부족

# 메모리 확인
docker stats supabase-postgres --no-stream

# CPU 확인
docker top supabase-postgres
\`\`\`

## 해결

### 해결책 1: 프로세스 재시작 (성공률 60%)
\`\`\`bash
docker-compose restart postgres

# 5초 대기
sleep 5

# 확인
docker exec supabase-postgres pg_isready -U postgres
# accepting connections ← 이 출력이 나올 때까지 기다리기
\`\`\`

### 해결책 2: 디스크 정리 (필요시)
\`\`\`bash
# 로그 파일 확인
du -sh /var/lib/docker/volumes/*

# 오래된 백업 정리
ls -lh /backup/ | tail -10
rm /backup/supabase-backup-*.sql.gz  # 7일 이상 된 파일만
\`\`\`

### 해결책 3: 컨테이너 강제 재시작 (성공률 80%)
\`\`\`bash
docker-compose down postgres
sleep 10
docker-compose up -d postgres

# 다시 시작할 때까지 기다리기 (보통 30초)
sleep 30

# 재확인
docker exec supabase-postgres pg_isready -U postgres
\`\`\`

### 해결책 4: 데이터 확인 및 복구
\`\`\`bash
# 데이터 무결성 확인
docker exec supabase-postgres psql -U postgres \
  -c "REINDEX DATABASE postgres;"

# 마지막 백업에서 복구 (최후의 수단)
# /backup/supabase-*.sql.gz 파일 있는지 확인
ls /backup/ | grep supabase

# 복구 실행 (데이터 손실 가능 - 신중!)
docker exec -i supabase-postgres psql -U postgres < /backup/supabase-latest.sql
\`\`\`

## 검증

\`\`\`bash
# 1. 기본 연결 확인
docker exec supabase-postgres psql -U postgres -c "SELECT 1;"
# 출력: 1 ← 성공

# 2. 데이터 조회 테스트
docker exec supabase-postgres psql -U postgres -c \
  "SELECT COUNT(*) FROM information_schema.tables;"

# 3. 애플리케이션 API 테스트
curl http://localhost:8000/rest/v1/ -H "Authorization: Bearer test"
# HTTP 200 응답 ← 성공

# 4. 모니터링 확인
curl http://localhost:3001/api/health | jq .database
# "status": "ok" ← 성공
\`\`\`

## Escalation

- **자동 재시작으로 복구**: 팀에 공지 (Slack #incident)
- **수동 재시작 필요**: 팀 리드에 전화 알림
- **복구 불가 (30분 이상)**: CTO 호출, 고객 공지 준비

## Post-mortem 항목

```
[ ] 정확한 다운 시작 시간?
[ ] 몇 명의 사용자 영향?
[ ] 근본 원인?
[ ] 왜 사전 모니터링에 안 걸렸나?
[ ] 같은 일을 막으려면?
```

### **장애 2: API 서버 502 Bad Gateway**

```markdown
# [Runbook] API 502 에러 (Bad Gateway)

## 증상
- 브라우저: "502 Bad Gateway"
- 클라이언트: API 요청 실패
- Status Page: API DOWN

## 영향도
- **범위**: 모든 REST API 요청
- **중요도**: CRITICAL

## 원인 추론
1. REST API 서비스 다운
2. API 프로세스 충돌 (메모리 부족)
3. API ↔ PostgreSQL 연결 끊김
4. 역 프록시(Caddy) 설정 오류

## 진단

\`\`\`bash
# 1. API 컨테이너 상태
docker ps | grep api

# 2. API 로그 확인
docker logs supabase-api --tail=50

# 3. Caddy 상태 확인 (역 프록시)
docker logs caddy --tail=50

# 4. 포트 점유 확인
netstat -an | grep 8000
# 또는
ss -tlnp | grep 8000
\`\`\`

## 해결

\`\`\`bash
# 해결책 1: API 재시작
docker-compose restart rest-api

# 해결책 2: Caddy 재시작 (역 프록시)
docker-compose restart caddy

# 해결책 3: 포트 점유 프로세스 강제 종료
lsof -i :8000 | tail -1 | awk '{print $2}' | xargs kill -9
sleep 5
docker-compose up -d rest-api

# 해결책 4: 메모리 부족 확인
docker stats rest-api --no-stream
# MEMORY 컬럼에서 1GB 이상이면 재시작 필요
\`\`\`

## 검증

\`\`\`bash
curl -i http://localhost:8000/health
# HTTP/1.1 200 OK ← 성공

# 실제 API 요청 테스트
curl http://localhost:8000/rest/v1/users \
  -H "Authorization: Bearer $TOKEN"
\`\`\`

### **장애 3: 디스크 용량 부족**

```markdown
# [Runbook] 디스크 용량 부족 (Disk Full)

## 증상
- 로그에 "No space left on device"
- 파일 업로드 실패
- 데이터베이스 쓰기 실패

## 영향도
- **범위**: 파일 업로드, DB 쓰기 작업
- **중요도**: HIGH (즉시 대응)

## 진단

\`\`\`bash
# 1. 전체 디스크 상태
df -h /

# 예상 출력:
# Size  Used Avail Use%
# 100G  95G  5G   95%  ← 위험! (80% 이상)

# 2. 주요 디렉토리 크기
du -sh /var/lib/docker/*
du -sh /backup/*
du -sh /home/*

# 3. 가장 큰 파일 찾기
find / -type f -size +100M -exec du -sh {} + | sort -rh | head -20
\`\`\`

## 해결

\`\`\`bash
# 해결책 1: 오래된 로그 삭제
docker logs supabase-postgres --timestamps | head -1
# 로그가 10GB 이상이면:
docker logs supabase-postgres > /tmp/old-logs.txt  # 저장
# 컨테이너 재시작으로 로그 초기화
docker-compose restart postgres

# 해결책 2: 오래된 백업 삭제
ls -lh /backup/ | sort -k6
# 7일 이상 된 백업 삭제
find /backup -name "supabase-*.sql.gz" -mtime +7 -delete

# 해결책 3: Docker 이미지 정리
docker image prune -a  # 사용하지 않는 이미지 삭제

# 해결책 4: 임시 파일 정리
rm -rf /tmp/*
rm -rf /var/tmp/*

# 마지막 확인
df -h /
# Use% < 80% ← 정상
\`\`\`

### **장애 4: SSL 인증서 만료**

```markdown
# [Runbook] SSL 인증서 만료

## 증상
- 브라우저: "Your connection is not secure"
- API 호출: "certificate has expired"
- HTTPS 접속 불가

## 영향도
- **범위**: 모든 HTTPS 연결
- **중요도**: CRITICAL

## 진단

\`\`\`bash
# 현재 인증서 확인
docker exec caddy caddy list-certs

# 또는 Cloudflare에서 확인
curl -s https://api.your-domain.com/ -I | grep -i date

# 또는 명령어로
openssl s_client -connect api.your-domain.com:443 \
  -showcerts 2>/dev/null | grep -A1 "Validity"
\`\`\`

## 해결

\`\`\`bash
# Caddy는 자동 갱신 (Let's Encrypt)
# → 보통 자동 해결됨

# 수동 갱신이 필요한 경우:
docker exec caddy caddy renew

# Caddy 재시작
docker-compose restart caddy

# 확인
curl -i https://api.your-domain.com/health
# HTTP/1.1 200 OK
\`\`\`

### **장애 5: Real-time 연결 끊김**

```markdown
# [Runbook] Real-time (WebSocket) 연결 끊김

## 증상
- 채팅/알림이 실시간으로 안 옴
- 클라이언트: "WebSocket connection failed"
- Real-time 서버 CPU 100%

## 영향도
- **범위**: 실시간 기능만
- **중요도**: MEDIUM (기능 제한)

## 진단

\`\`\`bash
# Real-time 서버 상태
docker ps | grep realtime

# 로그 확인
docker logs supabase-realtime --tail=50

# 메모리/CPU 확인
docker stats supabase-realtime --no-stream

# 포트 확인
netstat -an | grep 8001
\`\`\`

## 해결

\`\`\`bash
# 해결책 1: Real-time 재시작
docker-compose restart realtime
sleep 10

# 해결책 2: WebSocket 포트 확인
curl -i ws://localhost:8001/ws
# 또는
curl -v http://localhost:8001/health

# 해결책 3: 메모리 확인 (누수 가능)
docker stats supabase-realtime --no-stream
# Memory > 500MB이면 재시작 필요
\`\`\`

### 5. Post-mortem 템플릿

```markdown
# Post-mortem: [장애명] (2026-04-01)

## 1. 사건 요약
- **발생 시간**: 2026-04-01 14:30 UTC
- **복구 시간**: 2026-04-01 14:45 UTC
- **영향도**: 500명 사용자, 15분
- **심각도**: HIGH

## 2. 사건 타임라인

| 시간 | 이벤트 |
|------|--------|
| 14:30 | 사용자 민원 첫 보고 |
| 14:32 | 팀 슬랙 알림, 진단 시작 |
| 14:38 | 근본 원인 파악 (DB 메모리 부족) |
| 14:40 | 임시 해결 (컨테이너 재시작) |
| 14:45 | 정상화 확인 |

## 3. 근본 원인 분석 (RCA)

**직접 원인 (Immediate Cause):**
- PostgreSQL 메모리 누수로 OOM
- 자동 재시작 설정 미흡

**근본 원인 (Root Cause):**
- 쿼리 성능 저하로 메모리 사용 급증
- 특정 쿼리에 INDEX 없음
- 모니터링 임계값 설정이 느슨함

## 4. 개선 방안

| 항목 | 현황 | 개선안 | 담당 | 완료일 |
|------|------|--------|------|---------|
| 쿼리 최적화 | 미흡 | INDEX 추가 | 개발팀 | 2026-04-05 |
| 메모리 모니터링 | 임계값 고정 | 자동 스케일 | 운영팀 | 2026-04-08 |
| 자동 재시작 | 미설정 | Docker health check 추가 | 운영팀 | 2026-04-03 |

## 5. 예방

- [ ] INDEX 추가 완료
- [ ] 모니터링 임계값 조정
- [ ] 자동 헬스 체크 활성화
- [ ] 팀 교육 (Runbook 검토)

---
```
