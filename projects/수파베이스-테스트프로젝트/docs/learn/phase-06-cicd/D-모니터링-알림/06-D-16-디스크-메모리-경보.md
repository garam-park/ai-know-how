**다음 학습**: Supabase 버전 확인 & 업그레이드 전략 (그룹 E)

## 개요

Grafana **Alert Rules**를 설정하면 디스크가 80% 찼을 때, 메모리가 85% 도달했을 때 자동으로 Discord/Slack으로 알림을 보냅니다.

Uptime Kuma는 "서비스 다운?"을 감지하고, Grafana 경보는 "리소스 부족 경고"를 사전에 알립니다.

### FE 개발자를 위한 비유

| 개념 | JavaScript 이벤트 | Grafana 경보 |
|------|------------|----------|
| **데이터 감시** | `addEventListener('input')` | PromQL 쿼리로 메트릭 감시 |
| **조건 판단** | `if (value > threshold)` | Alert Rule: 조건식 평가 |
| **액션 트리거** | `callback()` 실행 | 알림 채널로 POST 요청 |
| **상태 관리** | 이벤트 발생 1회 | Alerting, Firing, Resolved 상태 |

## Step 1: Grafana 알림 채널 설정

### 방법 1: Discord 알림 채널

#### Discord 웹훅 생성 (06-D-13과 동일)

1. Discord 서버 → #monitoring 채널 → Settings → Integrations
2. Webhooks → New Webhook
3. "Grafana Alerts" 이름 설정
4. URL 복사

#### Grafana에 Discord 채널 추가

1. 좌측 메뉴 → **Alerting** → **Contact Points** 클릭
2. **+ Create contact point** 클릭
3. 설정:
   ```
   Name: Discord
   Integration: Discord
   Webhook URL: [위에서 복사한 URL]
   Title: Supabase Alert
   Message template: (기본값)
   ```
4. **Test** 클릭 → Discord #monitoring에 테스트 메시지 확인
5. **Save contact point** 클릭

### 방법 2: Slack 알림 채널 (선택)

#### Slack 웹훅 생성

1. [Slack API Dashboard](https://api.slack.com/apps) → 기존 앱 또는 새 앱
2. Incoming Webhooks 활성화
3. Add New Webhook → #monitoring 채널 선택
4. URL 복사

#### Grafana에 Slack 채널 추가

1. **Alerting** → **Contact Points** → **+ Create contact point**
2. 설정:
   ```
   Name: Slack
   Integration: Slack
   Webhook URL: [복사한 URL]
   ```
3. **Test** → Slack 확인
4. **Save contact point** 클릭

## Step 2: 알림 규칙(Notification Policy) 설정

1. **Alerting** → **Notification Policies** 클릭
2. **Edit policy** (기본 정책)
3. 설정:
   ```
   Default contact point: Discord (또는 Slack)
   Group by: alertname, instance
   Group wait: 10s (경보 그룹화 대기)
   Group interval: 5m (그룹 반복 간격)
   Repeat interval: 4h (해결되지 않은 경보 재알림 간격)
   ```
4. **Save policy** 클릭

이 설정은 **비슷한 경보들을 묶어서 한 번에 보냄** (경보 폭주 방지).

## Step 3: 디스크 사용률 경보 규칙 생성

### 규칙 생성

1. **Alerting** → **Alert rules** → **+ Create alert rule** 클릭
2. 설정:

```
Rule name: Disk Usage > 80%
Folder: Alerts
Group: System

Condition (매 1분마다 평가):
  Query A: node_filesystem_avail_bytes{fstype!~"tmpfs|squashfs"}
  Reduce: Last value
  Math: 100 - $A

  OR manually set:
  node_filesystem_avail_bytes / node_filesystem_size_bytes < 0.2

Evaluate every: 1m
For: 5m (5분 동안 조건 충족 시 경보)
Alert state if execution error: Alerting
Alert state if no data: NoData

Annotations:
  summary: "Disk usage is high on {{ $labels.device }}"
  description: "Disk usage is {{ $value | humanizePercentage }} on {{ $labels.instance }}"

Labels:
  severity: critical
  environment: production
```

### 다른 표현 (더 간단함)

```promql
(1 - node_filesystem_avail_bytes{fstype!~"tmpfs|vfat|squashfs"}
     / node_filesystem_size_bytes) * 100 > 80
```

## Step 4: 메모리 사용률 경보 규칙 생성

### 규칙 생성

1. **+ Create alert rule** 클릭
2. 설정:

```
Rule name: Memory Usage > 85%
Folder: Alerts
Group: System

Condition:
  (1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) * 100 > 85

Evaluate every: 1m
For: 3m

Annotations:
  summary: "High memory usage"
  description: "Memory usage is {{ $value | humanizePercentage }} on {{ $labels.instance }}"

Labels:
  severity: warning
  environment: production
```

## Step 5: PostgreSQL 연결 경보 규칙 생성

### 규칙 생성

1. **+ Create alert rule** 클릭
2. 설정:

```
Rule name: Database Connection Count > 100
Folder: Alerts
Group: Database

Condition:
  pg_stat_activity_count > 100

Evaluate every: 1m
For: 2m

Annotations:
  summary: "High database connection count"
  description: "{{ $value }} active connections (threshold: 100)"

Labels:
  severity: critical
  component: database
```

## Step 6: CPU 사용률 경보 (선택)

### 규칙 생성

```
Rule name: CPU Usage > 80%
Folder: Alerts
Group: System

Condition:
  (1 - avg(rate(node_cpu_seconds_total{mode="idle"}[5m]))) * 100 > 80

Evaluate every: 1m
For: 5m

Annotations:
  summary: "High CPU usage: {{ $value | humanizePercentage }}"

Labels:
  severity: warning
```

## Step 7: 경보 테스트

### 테스트 1: 디스크 경보 테스트

#### 방법 1: 실제 테스트 파일 생성 (위험 ⚠️)

```bash
# 주의: 서버 용량을 실제로 채웁니다. 신중히 진행.
# 테스트용 더미 파일 생성 (1GB)
dd if=/dev/zero of=/tmp/test_file_1gb.bin bs=1M count=1024

# 경보 대기 (5분)
sleep 300

# 디스크 사용률 Grafana에서 모니터 - 80% 도달 시 경보 발동
# Discord/Slack 메시지 수신 확인

# 테스트 파일 삭제
rm /tmp/test_file_1gb.bin
```

#### 방법 2: 안전한 테스트 (권장)

```bash
# 1. Prometheus에서 직접 메트릭 확인
curl http://localhost:9090/api/v1/query?query='node_filesystem_avail_bytes'

# 2. PromQL 쿼리로 현재 디스크 상태 확인
# Prometheus UI (http://localhost:9090)에서:
(1 - node_filesystem_avail_bytes / node_filesystem_size_bytes) * 100

# 결과: 예) 45.3 (45.3% 사용 중)
```

### 테스트 2: 경보 규칙 평가 확인

1. Grafana → **Alerting** → **Alert rules**
2. 각 규칙 클릭 → **View recording rule** 또는 **Edit** 클릭
3. **Preview** 섹션에서 현재 값 확인

```
Rule: Disk Usage > 80%
Current Value: 45.3 (정상)

Rule: Memory Usage > 85%
Current Value: 72.1 (정상)
```

### 테스트 3: 강제 경보 발동 (테스트용)

1. **Alert rules** → 규칙 선택 → **Edit**
2. 임시로 threshold 낮춤:
   ```
   # 원본: > 80
   # 테스트: > 45 (현재 값보다 낮게)
   ```
3. 저장 후 약 5분 대기 (For: 5m)
4. Discord/Slack 메시지 수신 확인
5. 다시 원래 값으로 수정

## Step 8: 경보 상태 모니터링

### 경보 상태 종류

```
┌─────────────────────────────────────┐
│ Normal (정상)                        │
├─────────────────────────────────────┤
│ 조건: 메트릭 < threshold              │
│ 동작: 아무 일 없음                     │
└─────────────────────────────────────┘
                ↓
┌─────────────────────────────────────┐
│ Alerting (조건 충족, 대기 중)          │
├─────────────────────────────────────┤
│ 조건: 메트릭 > threshold               │
│ 동작: For 시간 동안 대기               │
│      (예: 5분 동안 유지)               │
└─────────────────────────────────────┘
                ↓
┌─────────────────────────────────────┐
│ Firing (경보 발동)                    │
├─────────────────────────────────────┤
│ 조건: For 기간 만료 후도 조건 충족     │
│ 동작: Discord/Slack 알림 발송         │
│      repeat_interval마다 반복         │
└─────────────────────────────────────┘
```

### Grafana UI에서 경보 상태 확인

1. **Alerting** → **Alert instances** 클릭
2. 현재 firing/pending/normal 상태 표시
3. 각 경보 클릭 → 상세 정보:
   ```
   Rule: Disk Usage > 80%
   State: Normal
   Current value: 45.3%
   Evaluation: Last evaluated 1m ago
   ```

## Step 9: 경보 조정 및 최적화

### 잘못된 경보 (False Positive) 방지

| 현상 | 원인 | 해결책 |
|------|------|--------|
| 경보가 너무 자주 울림 | For 시간 너무 짧음 | For: 5m → For: 10m 증가 |
| 경보가 안 울림 | Threshold 너무 높음 | 80% → 75%로 낮춤 |
| 자정 경보만 울림 | 특정 시간만 높음 | 시간 범위 필터링 (고급) |

### 심각도(Severity) 레이블 활용

```yaml
Labels 설정 시 심각도 구분:
- severity: critical  (즉시 대응)
- severity: warning   (곧 확인)
- severity: info      (참고)
```

Discord/Slack 메시지에 severity 표시됨.

### 경보 그룹화

Notification Policy에서:
```
Group by: alertname, instance, severity
```

비슷한 경보들을 하나의 메시지로 통합.

   ```
   Name: Email
   Integration: Email
   Addresses: your-team@company.com
   ```

## 트러블슈팅

### 경보가 발동하지 않음

1. **규칙이 실제로 평가되는지 확인**
   ```bash
   curl http://localhost:9090/api/v1/rules | jq '.data.groups[]'
   ```

2. **PromQL 쿼리 문법 오류 확인**
   - Prometheus UI에서 쿼리 테스트

3. **데이터 부족**
   ```promql
   # 메트릭이 존재하는지 확인
   node_filesystem_avail_bytes
   pg_stat_activity_count
   ```

### Discord/Slack 메시지 안 옴

1. **웹훅 URL 확인**
   ```bash
   curl -X POST https://discordapp.com/api/webhooks/YOUR_URL \
     -H "Content-Type: application/json" \
     -d '{"content":"Test"}'
   ```

2. **Grafana 로그 확인**
   ```bash
   docker-compose logs grafana | grep -i "alert\|webhook"
   ```

### 경보가 너무 자주 울림

1. For 시간 증가: `For: 5m` → `For: 10m`
2. 또는 Threshold 조정: `80%` → `85%`
3. Repeat interval 증가: `4h` → `8h`

## 핵심 요약

| 항목 | 설명 |
|------|------|
| **Alert Rule** | PromQL 조건식 + 기간 + 라벨 + 주석 |
| **평가 주기** | 기본 1분 (Evaluate every) |
| **발동 조건** | 조건 충족 시간 >= For 기간 |
| **상태** | Alerting (조건 충족) → Firing (발동) → Resolved (해결) |
| **연락처** | Contact Point: Discord, Slack, Email 등 |
| **그룹화** | Notification Policy로 비슷한 경보 묶음 |
| **심각도** | severity 레이블로 우선순위 표시 |
| **재알림** | repeat_interval (기본 4시간)마다 반복 |
