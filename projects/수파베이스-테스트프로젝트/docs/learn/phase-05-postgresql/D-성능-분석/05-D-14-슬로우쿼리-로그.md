# 05-D-14 슬로우 쿼리 로그 활성화 및 확인

## 이 파일에서 하는 것

PostgreSQL 슬로우 쿼리 로그를 활성화하여 느린 쿼리를 발견하고 분석하는 방법을 학습합니다.

> **FE 개발자를 위한 비유**: Chrome DevTools의 Performance 탭에서 느린 API 호출이나 렌더링을 찾는 것처럼, 슬로우 쿼리 로그는 **서버 사이드 Performance 프로파일러**입니다. 프론트엔드에서 느린 API 응답을 감지했다면, 데이터베이스 쿼리가 병목인지 확인할 수 있습니다.

---

## 사전 조건

- Supabase self-hosted가 실행 중인 상태
- superuser 권한으로 psql 접속 가능
- Docker 환경에서 PostgreSQL 컨테이너 접근 가능

---

## Step 1: 현재 슬로우 쿼리 설정 확인

### 목표
PostgreSQL의 현재 슬로우 쿼리 로깅 설정 상태를 파악합니다.

### 실행 과정

**1단계: psql로 접속**

```bash
docker exec -it supabase-db psql -U postgres
```

**2단계: 현재 설정 확인**

```sql
SHOW log_min_duration_statement;
```

#### 예상 결과

```
 log_min_duration_statement
---------------------------
 -1
(1 row)
```

| 값 | 의미 |
|---|---|
| `-1` | 슬로우 쿼리 로깅 비활성화 (모든 쿼리 실행하지만 로깅 안 함) |
| `0` | 모든 쿼리 로깅 (개발 환경에서만 사용) |
| `1000` | 1초 이상 걸리는 쿼리만 로깅 |
| `5000` | 5초 이상 걸리는 쿼리만 로깅 |

### 완료 기준

- [ ] `SHOW log_min_duration_statement;` 실행 성공
- [ ] 현재 설정값 확인됨

---

## Step 2: 슬로우 쿼리 설정 변경

### 목표
PostgreSQL이 1초 이상 걸리는 쿼리를 로그에 기록하도록 설정합니다.

### 실행 과정

#### 방법 A: ALTER SYSTEM 사용 (권장)

psql 내에서 다음 명령어를 실행합니다:

```sql
ALTER SYSTEM SET log_min_duration_statement = 1000;
ALTER SYSTEM SET log_statement = 'none';
```

**설명**:
- `log_min_duration_statement = 1000`: 1초(1000ms) 이상 걸리는 쿼리만 로깅
- `log_statement = 'none'`: 모든 쿼리 로깅 비활성화 (슬로우 쿼리만 로깅하기 위해)

**결과 메시지**:

```
ALTER SYSTEM
```

> **Docker 환경 주의**: ALTER SYSTEM으로 변경한 설정은 컨테이너를 재생성(docker compose down && up)하면 초기화될 수 있습니다. 영구 설정을 원하면 docker-compose.yml의 PostgreSQL 서비스에 command 옵션으로 추가하세요:
> ```yaml
> command: postgres -c log_min_duration_statement=1000
> ```

#### 방법 B: Docker 환경에서 직접 설정 (대안)

먼저 postgresql.conf 파일 위치 확인:

```bash
docker exec supabase-db find /var/lib/postgresql -name "postgresql.conf" 2>/dev/null
```

파일을 직접 수정:

```bash
docker exec supabase-db sed -i 's/#log_min_duration_statement = -1/log_min_duration_statement = 1000/' /var/lib/postgresql/data/postgresql.conf
```

### 완료 기준

- [ ] `ALTER SYSTEM SET` 명령어 실행 성공 (또는 파일 수정 완료)
- [ ] 오류 메시지 없음

## Step 3: 설정 리로드

### 목표
PostgreSQL에게 새로운 설정을 적용하도록 신호를 보냅니다.

### 실행 과정

psql 내에서 다음 명령어를 실행:

```sql
SELECT pg_reload_conf();
```

#### 예상 결과

```
 pg_reload_conf
----------------
 t
(1 row)
```

**의미**: `t` (true)는 리로드 성공을 의미합니다.

### 완료 기준

- [ ] `SELECT pg_reload_conf();` 실행 성공
- [ ] 반환값이 `t` (true)

## Step 4: 테스트용 슬로우 쿼리 실행

### 목표
슬로우 쿼리 로깅이 제대로 작동하는지 확인합니다.

### 실행 과정

**1단계: 테스트 쿼리 실행**

psql 내에서:

```sql
SELECT pg_sleep(2);
```

**의미**: 2초 동안 대기하는 쿼리입니다.

**2단계: 빠른 쿼리도 실행**

```sql
SELECT 1;
```

**의미**: 즉시 반환되는 쿼리 (로그에 기록되지 않아야 함)

### 완료 기준

- [ ] `SELECT pg_sleep(2);` 실행 완료 (약 2초 소요)
- [ ] `SELECT 1;` 실행 완료 (즉시 반환)
- [ ] psql에서 오류 없음

---

## Step 5: 로그 확인 방법

### 목표
실행한 슬로우 쿼리가 로그에 기록되었는지 확인합니다.

### 실행 과정

#### 방법 A: Docker 로그 확인 (권장)

```bash
docker logs supabase-db | grep "duration:"
```

#### 예상 결과

```
2026-04-01 10:23:45.123 UTC [1234] LOG:  duration: 2001.234 ms  statement: SELECT pg_sleep(2);
```

> **Supabase Docker 환경**: Supabase 공식 이미지는 기본적으로 stderr로 로그를 출력합니다. `docker logs supabase-db` 명령이 가장 확실한 로그 확인 방법입니다. 파일 기반 로그 경로(`/var/lib/postgresql/data/log/`)는 별도 설정이 필요합니다.

#### 방법 B: PostgreSQL 로그 파일 확인

Docker 컨테이너 내부 로그 파일:

```bash
docker exec supabase-db tail -f /var/lib/postgresql/data/log/postgresql-*.log
```

#### 로그 해석

```
duration: 2001.234 ms
↓
쿼리 실행 시간 (ms 단위)

statement: SELECT pg_sleep(2);
↓
실행된 SQL 쿼리
```

| 필드 | 설명 |
|---|---|
| `LOG:` | 로그 레벨 |
| `duration: 2001.234 ms` | 쿼리 실행 시간 (밀리초) |
| `statement:` | 실행된 SQL 쿼리 |

### 완료 기준

- [ ] `docker logs supabase-db | grep "duration:"` 실행
- [ ] 슬로우 쿼리 로그 확인됨 (최소 1개 이상)
- [ ] `SELECT pg_sleep(2);` 쿼리가 로그에 기록됨
- [ ] `SELECT 1;` 쿼리는 로그에 없음 (1초 미만)

## Step 6: 로그 분석 팁

### 로그 필터링

#### 가장 느린 쿼리 찾기

```bash
docker logs supabase-db | grep "duration:" | sort -t: -k2 -rn | head -5
```

#### 특정 패턴의 쿼리 찾기

```bash
docker logs supabase-db | grep "duration:" | grep "SELECT"
```

#### 최근 100줄 로그 확인

```bash
docker logs supabase-db | tail -100 | grep "duration:"
```

### 로그 데이터 추출

**더 읽기 쉬운 형식으로 변환**:

```bash
docker logs supabase-db 2>&1 | grep "duration:" | awk '{
  for(i=1;i<=NF;i++) {
    if($i ~ /^duration:/) print $i, $(i+1), $(i+2)
  }
}' | sort -rn
```

### 성능 분석 기준

| 쿼리 시간 | 평가 | 조치 |
|---|---|---|
| < 10ms | 매우 빠름 | 모니터링 불필요 |
| 10~100ms | 양호 | 필요시 최적화 고려 |
| 100~1000ms | 느림 | 인덱스 추가, 쿼리 최적화 필요 |
| > 1000ms | 매우 느림 | 즉시 최적화 필요 |

## 확인 방법 체크리스트

### 설정 확인

- [ ] `SHOW log_min_duration_statement;` → `1000` 반환
- [ ] `SHOW log_statement;` → `'none'` 반환

**psql에서 실행**:

```sql
SHOW log_min_duration_statement;
SHOW log_statement;
```

### 로그 확인

- [ ] 슬로우 쿼리(≥1000ms) 로그 존재
- [ ] 빠른 쿼리(<1000ms) 로그 부재
- [ ] 로그 포맷 일관성 유지

**확인 명령어**:

```bash
docker logs supabase-db | grep "duration:" | wc -l
```

## 자주 겪는 오류

| 오류 | 원인 | 해결 방법 |
|---|---|---|
| `ERROR: unrecognized configuration parameter "log_min_duration_statement"` | PostgreSQL 버전 오류 | PostgreSQL 9.1 이상 확인 |
| 로그가 나타나지 않음 | 설정 미적용 또는 쿼리 시간 미달 | `pg_reload_conf()` 재실행, 쿼리 시간 확인 |
| `docker logs` 명령어 거부 | 권한 부족 | `sudo docker logs` 실행 또는 `sudo -s` |
| 로그 파일이 너무 빠르게 증가 | `log_min_duration_statement` 값이 너무 낮음 | 값을 1000ms 이상으로 증가 |
| `SELECT pg_reload_conf();` 결과가 `f` | 파일 오류로 인한 리로드 실패 | `postgresql.conf` 파일 문법 확인 |
