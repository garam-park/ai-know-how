# 04-C-14 컨테이너 첫 실행 & 상태 확인

> 유형: 실습 | 예상 소요 시간: 20~30분
> 이전: Supabase compose 파일 서버에 올리기 | 다음: Caddy 개념 & 설치

---

## 이 파일에서 하는 것

`docker compose up`으로 Supabase의 모든 서비스를 처음 실행합니다. FE 개발자 관점에서, `npm run dev`로 로컬 개발 서버를 여러 개 띄우는 것과 유사하지만, 각 서비스가 별도 프로세스(컨테이너)에서 독립적으로 실행됩니다. 각 서비스의 상태를 확인하고 문제를 진단하는 방법을 배웁니다.

---

## 사전 조건

- 04-C-12 Docker 설치 완료
- 04-C-13 docker-compose.yml과 .env 파일이 서버에 준비됨
- SSH로 서버에 접속한 상태

---

## 실습

### Step 1. 프로젝트 폴더로 이동

docker-compose.yml과 .env 파일이 있는 디렉토리로 이동합니다.

```bash
cd ~/supabase  # 또는 docker-compose.yml이 있는 경로
```

> 이 경로는 04-C-13에서 파일을 업로드한 위치입니다.

파일이 제대로 준비되었는지 확인합니다.

```bash
ls -la
```

예상 출력:
```
total 48
drwxr-xr-x  3 root root  4096 Apr  1 12:34 .
drwxr-xr-x  5 root root  4096 Apr  1 12:00 ..
-rw-r--r--  1 root root 28456 Apr  1 12:34 docker-compose.yml
-rw-r--r--  1 root root  2345 Apr  1 12:34 .env
drwxr-xr-x  2 root root  4096 Apr  1 12:34 volumes/
```

---

### Step 2. 서비스 실행 (백그라운드 모드)

모든 Supabase 서비스를 한 번에 시작합니다.

```bash
docker compose up -d
```

옵션 설명:
- `up`: 서비스 시작
- `-d`: 백그라운드 모드 (detached) — 실행 후 터미널 제어권이 반환됨

예상 출력:
```
[+] Running 12/12
 ✓ Container supabase-postgres-1                Started                   2.3s
 ✓ Container supabase-postgrest-1               Started                   3.5s
 ✓ Container supabase-auth-1                    Started                   4.1s
 ✓ Container supabase-realtime-1                Started                   4.5s
 ✓ Container supabase-storage-1                 Started                   5.2s
 ✓ Container supabase-vector-1                  Started                   5.8s
 ✓ Container supabase-kong-1                    Started                   6.2s
 ✓ Container supabase-studio-1                  Started                   7.0s
```

> 각 서비스가 순차적으로 시작되는 이유는 04-B-07에서 배운 의존성 때문입니다. postgres가 먼저 뜨고, 이에 의존하는 다른 서비스들이 차례로 기동됩니다.

---

### Step 3. 컨테이너 상태 확인

모든 컨테이너의 현재 상태를 확인합니다.

```bash
docker compose ps
```

예상 출력:
```
NAME                                    COMMAND                  SERVICE         STATUS              PORTS
supabase-postgres-1                     "docker-entrypoint.s…"   postgres        Up 2 minutes (healthy)    5432/tcp
supabase-vector-1                       "vector --config /et…"   vector          Up 2 minutes
supabase-postgrest-1                    "postgrest /etc/post…"   postgrest       Up 2 minutes (healthy)    3000/tcp
supabase-kong-1                         "/docker-entrypoint.…"   kong            Up 2 minutes (healthy)    0.0.0.0:8000->8000/tcp, 0.0.0.0:8443->8443/tcp
supabase-auth-1                         "auth"                   auth            Up 2 minutes (healthy)    9999/tcp
supabase-realtime-1                     "sh -c 'node /app/bui…"   realtime        Up 2 minutes (healthy)    4000/tcp
supabase-storage-1                      "/bin/sh -c 'npm run …"   storage         Up 2 minutes (healthy)    5000/tcp
supabase-studio-1                       "nginx -g 'daemon off…"   studio          Up 2 minutes (healthy)    0.0.0.0:3000->3000/tcp
```

**상태 해석:**

| 상태 | 의미 | 조치 |
|------|------|------|
| `Up X minutes (healthy)` | 정상 실행 중 | OK, 다음 단계 진행 |
| `Up X minutes (starting)` | 시작 중 | 잠깐 기다렸다가 재확인 |
| `Up X minutes (unhealthy)` | 실행되지만 문제 있음 | 로그 확인 필요 |
| `Exited (X)` | 비정상 종료 | 로그 확인하고 원인 파악 |
| `Container not found` | 컨테이너가 없음 | Step 2 다시 실행 |

---

### Step 4. 서비스 시작 순서 모니터링 (선택사항)

현재 상태는 완료된 상태입니다만, 처음 실행 시 각 서비스가 어떤 순서로 뜨는지 관찰하려면:

```bash
docker compose logs -f --tail=20
```

옵션 설명:
- `-f`: 실시간 로그 스트림 (follow) — 새로운 로그가 나오면 계속 표시
- `--tail=20`: 최근 20줄만 표시

예상 로그:
```
postgres-1     | 2026-04-01 12:35:10.123 UTC [1] LOG:  listening on IPv4 address "0.0.0.0", port 5432
postgres-1     | 2026-04-01 12:35:10.124 UTC [1] LOG:  database system is ready to accept connections
postgrest-1    | Listening on port 3000
auth-1         | [12:35:11] listening on http://0.0.0.0:9999
realtime-1     | [realtime] Listening on 0.0.0.0:4000
storage-1      | ▶ listening on 0.0.0.0:5000
kong-1         | 2026-04-01T12:35:15,263 [notice] 1#0: *1 client connected
studio-1       | 2026-04-01T12:35:16Z  INFO  nginx: config reload successful
```

Ctrl+C를 누르면 로그 스트림에서 나갑니다. (서비스는 계속 실행됨)

---

### Step 5. Kong 응답 확인 (API Gateway 작동 확인)

Kong이 요청을 받고 응답하는지 확인합니다. 이는 전체 API 진입점입니다.

```bash
curl http://localhost:8000
```

예상 출력:
```json
{
  "name":"kong",
  "description":"RESTful API for Kong admin",
  "version":"3.1.0",
  "hostname":"supabase-kong-1",
  "tagline":"So Fine, It Quit",
  "lua_version":"5.1.4"
}
```

이 응답이 나오면 Kong이 정상 작동하는 것입니다.

> 404 또는 `Connection refused` 에러가 나면 Kong 컨테이너를 확인하세요. (Step 3의 상태 확인)

---

### Step 6. PostgREST REST API 확인

PostgREST가 REST API를 제공하는지 확인합니다. (실제 데이터 쿼리는 하지 않고 API 접근만 테스트)

```bash
curl -i http://localhost:8000/rest/v1/
```

예상 출력:
```
HTTP/1.1 200 OK
...
{"swagger":"2.0","info":{"title":"PostgREST API","version":"12.0.0"},...}
```

`200 OK` 응답이 나오면 PostgREST가 정상 작동하는 것입니다.

> 404나 502 Bad Gateway가 나면 postgres나 postgrest 컨테이너를 확인하세요.

---

### Step 7. 각 서비스별 상태 확인 포인트

각 서비스가 뭘 기다리는지, 어떤 순서로 뜨는지 이해하면 문제 해결이 쉬워집니다.

**FE 비유:**
- `npm run dev`를 실행하면 Next.js, 타입스크립트 컴파일러, ESLint 등이 동시에 시작
- 하나가 실패하면 전체 서비스가 불안정해짐
- docker compose도 동일한 구조

**기동 순서:**
```
1. postgres (데이터베이스)
   ↓ (postgrest, auth, realtime 등이 postgres 기다림)
2. postgrest, auth, realtime, vector 동시 시작
   ↓ (kong이 postgrest, auth 기다림)
3. kong 시작 (API 라우터)
   ↓ (storage가 kong 기다림)
4. storage 시작
   ↓ (studio가 모두 기다림)
5. studio 시작 (관리 UI)
```

각 서비스별 로그 확인:

```bash
# postgres 로그만 보기
docker compose logs postgres

# postgrest 로그만 보기
docker compose logs postgrest

# kong 로그만 보기
docker compose logs kong

# 특정 서비스의 최근 50줄 실시간 보기
docker compose logs -f --tail=50 auth
```

---

### Step 8. Supabase Studio (관리 UI) 접속 확인

웹 브라우저에서 Studio에 접속할 수 있는지 확인합니다.

```
http://서버IP:3000
```

예상:
- Studio 로그인 화면이 보여야 함
- 초기 로그인 이메일/비밀번호는 04-C-13의 .env 파일에서 설정한 값

> localhost:3000이 아니라 서버의 실제 IP를 사용해야 합니다.
> 로컬 테스트를 위해 ssh 터널을 사용할 수 있습니다:
> ```bash
> ssh -L 3000:localhost:3000 user@server-ip
> ```
> 그 후 로컬에서 http://localhost:3000 접속

## 자주 겪는 오류와 해결

### 오류 1: `docker compose: command not found`

**원인:** Docker Compose가 설치되지 않음

**해결:**
```bash
sudo apt-get install docker-compose-plugin
```

---

### 오류 2: `Status: Exited (1)` 또는 `Status: Exited (127)`

**원인:** 컨테이너가 비정상 종료됨. 주로 환경 변수 오류, 이미지 다운로드 실패, 또는 포트 충돌.

**해결:**
```bash
# 해당 서비스의 로그 확인
docker compose logs postgres

# 전체 로그 확인
docker compose logs

# 컨테이너 재시작
docker compose restart postgres

# 모든 컨테이너 재시작
docker compose restart
```

**로그에서 보이는 일반적인 오류:**
- `could not translate host name "postgres" to address`: 네트워크 문제
- `FATAL: password authentication failed`: 비밀번호 오류 (.env 확인)
- `port is already in use`: 포트 충돌 (다른 서비스가 이미 사용 중)

---

### 오류 3: 메모리 부족 (`OOMKilled`)

**원인:** 서버 메모리가 부족함. 보통 Hetzner CX22(4GB)에서는 발생하지 않음.

**확인:**
```bash
docker compose ps
```

로그에 `OOMKilled` 또는 `Out of memory` 메시지가 있으면 메모리 부족.

**해결:**
- 서버 메모리 업그레이드 (Hetzner에서 CX31 이상으로 변경)
- 또는 불필요한 서비스 비활성화 (vector 등)

---

### 오류 4: `Unable to locate image` 또는 이미지 다운로드 실패

**원인:** Docker Hub에서 이미지를 다운로드할 수 없음. 보통 인터넷 연결 문제나 방화벽 차단.

**해결:**
```bash
# 인터넷 연결 확인
ping docker.io

# 또는 Docker Hub 미러 사용 (중국/일부 지역)
# docker-compose.yml의 image 이름 앞에 미러 URL 추가
# 예: image: docker.io/library/postgres:15 → image: mirrors.aliyun.com/library/postgres:15
```

---

### 오류 5: `.env 파일을 읽을 수 없음`

**원인:** .env 파일이 docker-compose.yml과 같은 디렉토리에 없음.

**확인:**
```bash
ls -la | grep .env
```

**.env 파일이 보이지 않으면:**
```bash
# 04-C-13에서 업로드한 경로 확인
pwd
cd ~/supabase  # 또는 올바른 경로로 이동
```

---

### 오류 6: Kong이 `unhealthy` 상태

**원인:** kong이 postgrest나 auth와 통신하지 못함.

**확인:**
```bash
docker compose logs kong
```

로그에 `upstream service unavailable` 또는 `connection refused` 메시지가 있으면:

```bash
# postgrest와 auth 상태 확인
docker compose ps postgrest auth

# 재시작
docker compose restart postgrest auth kong
```

---

### 오류 7: `curl: (7) Failed to connect to 127.0.0.1:8000`

**원인:** Kong이 포트 8000에서 수신하지 않음.

**확인:**
```bash
# Kong 컨테이너 상태
docker compose ps kong

# Kong 로그
docker compose logs kong

# 포트 확인 (8000이 이미 사용 중인지 확인)
sudo lsof -i :8000
# 또는
sudo netstat -tlnp | grep 8000
```

다른 프로세스가 포트 8000을 사용 중이면:
- docker-compose.yml에서 Kong의 포트 번호 변경
- 또는 기존 프로세스 종료

## 핵심 정리

- **`docker compose up -d`**: 모든 서비스를 백그라운드에서 시작
- **`docker compose ps`**: 현재 모든 컨테이너의 상태 확인
- **`docker compose logs -f`**: 실시간 로그 스트림 보기
- **서비스 기동 순서**: postgres → 의존하는 서비스들 → kong → 나머지
- **상태 확인**: `healthy` 상태가 정상, `unhealthy`나 `Exited`는 문제 신호
- **Kong 응답 확인**: `curl http://localhost:8000`으로 API Gateway 작동 확인
- **문제 해결**: 항상 로그를 먼저 확인 — `docker compose logs <service-name>`
