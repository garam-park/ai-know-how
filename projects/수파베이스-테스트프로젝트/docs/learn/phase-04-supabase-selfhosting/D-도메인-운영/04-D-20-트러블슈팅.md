# 04-D-20 자주 발생하는 오류 트러블슈팅

> 유형: 참고 | 필요할 때 열어보는 파일
> 관련 그룹: Phase 4 전체

---

## 언제 이 파일을 보는가

Supabase 셀프호스팅 환경에서 배포 후 서비스 운영 중에 다음과 같은 문제가 발생했을 때 참고합니다:
- 컨테이너가 계속 재시작되거나 unhealthy 상태
- API 호출 시 인증 오류
- SSL 인증서 또는 도메인 관련 문제
- 데이터베이스 연결 실패
- 네트워크 통신 문제

각 오류별로 진단 방법과 해결책을 단계적으로 따라하면 됩니다.

```bash
# 모든 컨테이너 상태 조회

docker compose logs --tail=50

# 특정 서비스 로그 확인
```

**환경 변수 확인**

**네트워크 연결성 확인**

```bash
# 특정 포트 열려있는지 확인 (localhost)
netstat -tuln | grep :PORT_NUMBER

# 컨테이너에서 외부 접근 테스트
```

### 1. 컨테이너 관련 오류

#### 1-1. 컨테이너가 계속 재시작됨 (Restart Loop)

**증상**
- `docker compose ps` 에서 특정 컨테이너의 STATUS가 `Restarting (exit 1)` 반복
- 몇 초마다 컨테이너가 죽었다가 다시 시작되는 무한 반복

**원인**
- 애플리케이션 시작 실패 (코드/설정 에러)
- 필수 의존성 서비스가 준비되지 않음
- 메모리 부족으로 인한 강제 종료
- 환경 변수 누락

**해결 방법**

1단계: 컨테이너의 전체 로그 확인 (재시작 이력 포함)
```bash
docker compose logs [service-name] --tail=100
```

2단계: 에러 메시지 분석 (보통 마지막 10-20줄에 핵심 정보)
```
예시 1) "panic: database connection failed"
  → 데이터베이스가 준비되지 않았거나 연결 정보 오류

예시 2) "error: address already in use"
  → 포트 충돌 (이미 다른 프로세스가 사용 중)

예시 3) "OOMKilled"
  → 메모리 부족 (다음 섹션 참고)
```

3단계: 의존성 서비스 상태 확인
```bash
# 데이터베이스 서비스 실행 중인지 확인
docker compose ps postgres

# 각 서비스가 준비될 때까지 대기 후 재시작
docker compose down
docker compose up -d postgres
# 15초 정도 대기
sleep 15
docker compose up -d
```

4단계: 환경 변수 재확인
```bash
# .env 파일의 필수 변수 리스트 확인
# - DATABASE_URL
# - JWT_SECRET
# - ANON_KEY
# - SERVICE_ROLE_KEY
# - SITE_URL (또는 domain 설정)

docker compose config | grep -A 20 "environment:"
```

#### 1-2. 특정 컨테이너만 Unhealthy

**증상**
- `docker compose ps` 에서 STATUS가 `Up (unhealthy)` 표시
- 정상 실행되지만 헬스 체크 실패

**원인**
- 헬스 체크 엔드포인트 응답 지연
- 의존성 서비스에 아직 준비되지 않음
- 내부 포트에서 헬스 체크 응답 불가

**해결 방법**

1단계: 헬스 체크 상태 상세 정보 확인
```bash
docker inspect [container-name] | grep -A 10 "Health"
```

2단계: 헬스 체크 엔드포인트 직접 테스트
```bash
# 컨테이너 내부에서 헬스 체크 엔드포인트 호출
docker compose exec [service-name] curl -v http://localhost:3000/health

# 또는 호스트에서 직접 호출
curl -v http://localhost:[EXPOSED_PORT]/health
```

3단계: 의존성 서비스 준비 완료 대기
```bash
# 데이터베이스 준비 완료까지 대기
docker compose exec postgres pg_isready -U postgres
# "accepting connections" 출력이 나올 때까지 대기

# 그 후 unhealthy 컨테이너 재시작
docker compose restart [service-name]
```

4단계: 이전 단계에서도 실패 시 전체 재시작
```bash
docker compose down --remove-orphans
docker compose up -d
```

#### 1-3. 이미지 Pull 실패

**증상**
```
ERROR: failed to pull image "supabase/supabase:..."
Error response from daemon: manifest not found
```

**원인**
- 잘못된 이미지 이름 또는 태그
- Docker Hub 접속 실패
- 이미지 저장소가 private이고 인증되지 않음
- 네트워크 연결 문제

**해결 방법**

1단계: `docker-compose.yml`의 이미지 태그 확인
```bash
# 현재 정의된 이미지 확인
grep "image:" docker-compose.yml | head -10

# 공식 Supabase 저장소에서 사용 가능한 태그 확인
# https://hub.docker.com/r/supabase/supabase/tags
```

2단계: 이미지 태그 업데이트 (필요 시)
```bash
# Supabase 공식 이미지 최신 버전으로 변경
# docker-compose.yml 수정:
# supabase/supabase:latest → supabase/supabase:1.x.x (특정 안정 버전)
```

3단계: Docker 데몬 재시작 및 다시 pull
```bash
# 기존 이미지 캐시 삭제
docker image prune -a

# 다시 시도
docker compose pull --no-parallel
```

4단계: 네트워크 확인 (방화벽/프록시)
```bash
# Docker 데몬에서 외부 접근 가능 확인
docker run --rm curlimages/curl curl -I https://hub.docker.com
```

#### 1-4. 메모리 부족 (OOM Killed)

**증상**
```
# 컨테이너 로그에서:
"killed" 또는 "OOMKilled"

# docker compose ps 에서:
STATUS: Exited (137)
```

**원인**
- 할당된 메모리 제한이 너무 낮음
- 애플리케이션이 메모리 누수
- 여러 무거운 서비스가 동시 실행

**해결 방법**

1단계: 현재 메모리 사용률 확인
```bash
# 호스트 전체 메모리 상태
free -h

# 각 컨테이너 리소스 사용률 실시간 모니터링
docker stats
```

2단계: 컨테이너별 메모리 제한 확인 및 증가
```bash
# docker-compose.yml 수정
services:
  [service-name]:
    deploy:
      resources:
        limits:
          memory: 512M    # 현재값
        reservations:
          memory: 256M    # 현재값

# 권장값 (프로덕션 기준):
# - postgres: 1G
# - kong: 512M
# - auth: 256M
# - rest: 256M
# - realtime: 256M
```

3단계: docker-compose.yml 수정 후 재시작
```bash
docker compose down
docker compose up -d

# 다시 메모리 사용률 모니터링
docker stats
```

4단계: 여전히 부족하면 호스트 메모리 확인
```bash
# 호스트에 할당된 물리 메모리 확인
dmidecode -t 17 | grep "Size"

# 또는 Docker Desktop / Docker VM 메모리 설정 확인
# Windows/Mac: Docker Desktop Settings > Resources > Memory
```

### 2. 인증/키 관련 오류

#### 2-1. 401 Unauthorized (API 호출 시)

**증상**
```
POST https://your-domain.com/rest/v1/[table]
Response: 401 Unauthorized
Body: {"message":"Unauthorized"}
```

**원인**
- Authorization 헤더 누락
- 잘못된 API 키 사용
- ANON_KEY와 SERVICE_ROLE_KEY 혼동
- JWT 토큰 만료 또는 서명 오류
- API 키가 .env에서 불러와지지 않음

**해결 방법**

1단계: 요청 헤더 확인 (클라이언트 코드)
```javascript
// ❌ 잘못된 예
const response = await fetch('https://your-domain.com/rest/v1/users');

// ✅ 올바른 예 (ANON_KEY 사용)
const response = await fetch('https://your-domain.com/rest/v1/users', {
  headers: {
    'Content-Type': 'application/json',
    'Authorization': 'Bearer YOUR_ANON_KEY',
  }
});

// ✅ 또는 Supabase 클라이언트 라이브러리 사용
import { createClient } from '@supabase/supabase-js';
const supabase = createClient(url, anonKey);
```

2단계: .env 파일에서 키 값 확인
```bash
# 현재 환경의 키 값 확인
grep -E "ANON_KEY|SERVICE_ROLE_KEY" .env

# 각 키의 형식 확인 (JWT 토큰 형식이어야 함)
# - eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9 로 시작해야 함
```

3단계: 키 생성 로직 재확인
```bash
# Supabase 초기화 시 자동 생성된 키 확인
docker compose logs supabase 2>&1 | grep -i "anon\|service"

# 또는 초기화 스크립트 재실행
docker compose exec supabase /bin/bash
# 안내 따라서 키 재생성
```

4단계: API 키 vs JWT 토큰 구분
```
용도별 키 사용:
- ANON_KEY (익명 사용자)
  : 클라이언트 사이드 (브라우저, 모바일)
  : 읽기/쓰기 권한 제한됨 (RLS 정책에 의존)

- SERVICE_ROLE_KEY (서버)
  : 백엔드 서버에서만 사용
  : 모든 권한 보유 (RLS 정책 우회)
  : 절대 클라이언트에 노출하면 안 됨
```

#### 2-2. JWT 관련 에러

**증상**
```
"Invalid token" 또는
"jwt malformed" 또는
"jwt expired"
```

**원인**
- JWT_SECRET 설정 오류
- 토큰의 서명이 서버 시크릿과 일치하지 않음
- 토큰이 만료됨
- 토큰 형식이 잘못됨

**해결 방법**

1단계: JWT_SECRET 확인
```bash
# .env에서 JWT_SECRET 값 확인
grep "JWT_SECRET" .env

# 길이 확인 (최소 32자 이상 권장)
grep "JWT_SECRET" .env | cut -d'=' -f2 | wc -c

# ❌ 너무 짧으면 안 됨 (예: "secret123")
# ✅ 충분히 강력해야 함 (예: 64자 이상의 랜덤 문자열)
```

2단계: JWT_SECRET 재생성 (필요 시)
```bash
# 새로운 강력한 시크릿 생성
openssl rand -base64 32

# .env에 업데이트
# JWT_SECRET=새로생성된값

# 변경 후 서비스 재시작
docker compose restart auth kong
```

3단계: 토큰 유효기간 확인
```bash
# 클라이언트에서 로그인 후 받은 토큰을 jwt.io 에서 디코딩
# https://jwt.io (온라인 도구)
# Payload의 "exp" 필드 확인: UNIX 타임스탬프
# 현재 시간 > exp 면 토큰 만료

# 현재 UNIX 타임스탐프 확인
date +%s
```

4단계: 토큰 갱신 로직 추가 (클라이언트)
```javascript
// Supabase 클라이언트에서 자동 갱신 설정
const supabase = createClient(url, anonKey, {
  auth: {
    autoRefreshToken: true,  // 자동 갱신 활성화
    persistSession: true,    // 세션 저장
  }
});

// 또는 수동으로 갱신
const { data, error } = await supabase.auth.refreshSession();
```

#### 2-3. ANON_KEY / SERVICE_ROLE_KEY 불일치

**증상**
```
클라이언트에서 ANON_KEY로 요청 시:
- 특정 테이블/행에만 접근 거부 (RLS 정책 문제 가능)

서버에서 SERVICE_ROLE_KEY로 요청 시:
- 모든 데이터에 접근 성공하지만 권한 문제 발생
```

**원인**
- 클라이언트에서 SERVICE_ROLE_KEY 사용 (보안 위험)
- 서버에서 ANON_KEY 사용 (권한 부족)
- RLS 정책이 ANON_KEY 권한을 제한하는데 정책 설정 오류
- 기존 키가 .env에 잘못 매핑됨

**해결 방법**

1단계: 코드에서 키 사용처 확인
```bash
# 프로젝트 전체에서 SERVICE_ROLE_KEY 사용처 확인
grep -r "SERVICE_ROLE_KEY" . --include="*.js" --include="*.ts"

# 결과에서 클라이언트 사이드 코드가 SERVICE_ROLE_KEY를 사용하면 안 됨
# ❌ 잘못된 예:
#   src/components/UserList.tsx 에서 SERVICE_ROLE_KEY 사용
# ✅ 올바른 예:
#   src/server/admin.js 에서 SERVICE_ROLE_KEY 사용
```

2단계: 초기화 시 키 선택
```
Supabase CLI 또는 콘솔에서:
1. 프로젝트 생성 시 API Keys 섹션으로 이동
2. anon (public) key 복사 → ANON_KEY로 설정
3. service_role key 복사 → SERVICE_ROLE_KEY로 설정
```

3단계: 정책 재설정 및 테스트
```sql
-- 데이터베이스에서 RLS 정책 확인
SELECT * FROM pg_policies;

-- 특정 테이블의 정책 확인
SELECT * FROM pg_policies WHERE tablename = 'users';

-- 정책 생성 예시:
ALTER TABLE users ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Public user data is readable"
  ON users FOR SELECT
  USING (true);  -- 모든 사용자 읽기 가능

CREATE POLICY "Users can update own data"
  ON users FOR UPDATE
  USING (auth.uid() = id);  -- 자신의 데이터만 수정
```

### 3. 네트워크/도메인 관련 오류

#### 3-1. SSL 인증서 발급 실패

**증상**
```
Let's Encrypt 인증서 발급 실패:
"Challenge failed" 또는
"acme: error code 400"

브라우저 경고:
"NET::ERR_CERT_AUTHORITY_INVALID"
```

**원인**
- 도메인이 DNS에 아직 전파되지 않음
- 포트 80/443이 외부에서 접근 불가
- 방화벽이 ACME 챌린지 요청 차단
- 인증서 발급 엔드포인트의 임시 문제

**해결 방법**

1단계: DNS 전파 확인
```bash
# 도메인의 DNS 레코드 확인 (여러 방법)

# 방법 1: nslookup 사용
nslookup your-domain.com 8.8.8.8

# 방법 2: dig 사용
dig your-domain.com

# 방법 3: 온라인 도구
# https://mxtoolbox.com/
# https://dns.google/ 에서 조회

# ✅ 성공: 호스트 IP가 표시됨
# ❌ 실패: NXDOMAIN 또는 타임아웃

# DNS 전파 확인 사이트 (글로벌)
# https://whatsmydns.net/
# 전 세계 DNS 서버에서 도메인이 제대로 전파되었는지 확인
```

2단계: 포트 접근성 확인
```bash
# 호스트에서 포트 80/443이 외부에서 접근 가능한지 확인

# 방법 1: 로컬에서 테스트
curl -I http://localhost/
curl -I https://localhost/

# 방법 2: 원격에서 테스트 (또 다른 서버/VPN 사용)
curl -I http://your-domain.com/

# 방법 3: 포트 스캔 도구 (온라인)
# https://www.yougetsignal.com/tools/open-ports/
```

3단계: 방화벽 규칙 확인
```bash
# Linux 호스트의 방화벽 확인

# UFW 사용 중
sudo ufw status
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# iptables 사용 중
sudo iptables -L -n | grep -E "80|443"

# 또는 클라우드 환경 (AWS, GCP, Azure)
# 보안 그룹 / 네트워크 ACL 확인 후 포트 80, 443 허용 설정
```

4단계: Nginx/Reverse Proxy 설정 확인
```bash
# docker-compose.yml 의 nginx 또는 proxy 서비스 설정 확인
docker compose logs proxy --tail=50

# HTTP를 HTTPS로 리다이렉트하는 설정 확인
docker compose exec proxy cat /etc/nginx/conf.d/default.conf | grep -A 5 "listen 80"

# 예상 설정:
server {
  listen 80;
  server_name your-domain.com;
  return 301 https://$server_name$request_uri;
}
```

5단계: Let's Encrypt 재시도
```bash
# Certbot 로그 확인
docker compose logs certbot --tail=100

# 또는 서비스 재시작 (자동 갱신)
docker compose restart certbot

# 인증서 수동 발급 (필요 시)
docker compose exec certbot certbot certonly \
  --webroot \
  -w /var/www/certbot \
  -d your-domain.com \
  -d www.your-domain.com \
  --agree-tos \
  -m your-email@example.com
```

#### 3-2. CORS 에러

**증상**
```
브라우저 콘솔:
"Access to XMLHttpRequest at 'https://api.your-domain.com/...'
from origin 'https://your-domain.com'
has been blocked by CORS policy"
```

**원인**
- Supabase CORS 설정에 도메인 미등록
- Nginx/Proxy 응답 헤더 누락
- 요청 메서드가 CORS 정책에서 허용되지 않음

**해결 방법**

1단계: Supabase REST API 설정 확인
```bash
# docker-compose.yml 에서 REST 서비스의 CORS 설정 확인
grep -A 10 "rest:" docker-compose.yml

# PGRST_JWT_SECRET 및 기본 설정 확인
grep "PGRST" .env
```

2단계: Nginx 응답 헤더 설정
```bash
# Nginx 설정 파일 확인
docker compose exec proxy cat /etc/nginx/conf.d/default.conf

# CORS 헤더 추가 (필요 시):
# add_header 'Access-Control-Allow-Origin' '$http_origin';
# add_header 'Access-Control-Allow-Credentials' 'true';
# add_header 'Access-Control-Allow-Methods' 'GET, POST, PUT, DELETE, OPTIONS';
# add_header 'Access-Control-Allow-Headers' 'Content-Type, Authorization';
```

3단계: OPTIONS 프리플라이트 요청 처리 확인
```bash
# 브라우저 DevTools에서 Network 탭으로 확인
# 1. OPTIONS 요청이 성공 (200) 해야 함
# 2. 응답 헤더에 Access-Control-Allow-* 포함되어야 함

# 수동 테스트
curl -X OPTIONS https://your-domain.com/rest/v1/users \
  -H "Origin: https://your-domain.com" \
  -H "Access-Control-Request-Method: POST" \
  -v

# ✅ 성공: 200 응답 + CORS 헤더
# ❌ 실패: 4xx/5xx 또는 CORS 헤더 없음
```

4단계: 최후의 수단 - 프록시 우회 (테스트 용도만)
```javascript
// ❌ 프로덕션에서는 사용 금지
// 테스트 중에만 임시로 CORS 비활성화:

// 브라우저 개발자 도구 사용 (개발 환경)
// Chrome: --disable-web-security 플래그로 실행 (테스트 용도)
```

#### 3-3. 502 Bad Gateway

**증상**
```
브라우저:
"502 Bad Gateway"

클라이언트:
502 상태 코드 응답
```

**원인**
- 백엔드 서비스(REST, Auth 등)가 응답하지 않음
- Nginx/Proxy가 백엔드 서비스로 연결할 수 없음
- 백엔드 서비스가 crash 상태
- 네트워크 타임아웃

**해결 방법**

1단계: 백엔드 서비스 상태 확인
```bash
# 모든 서비스 상태 확인
docker compose ps

# REST API 서비스 상태 확인
docker compose ps | grep rest

# 상태가 "Exited" 이면 로그 확인
docker compose logs rest --tail=50
```

2단계: 네트워크 연결성 테스트 (Proxy에서)
```bash
# Proxy 컨테이너 내부에서 백엔드 서비스로 연결 테스트
docker compose exec proxy curl -v http://rest:3000/

# 성공 응답 (예: 200 또는 404) 이면 서비스 정상
# Connection refused 또는 타임아웃이면 서비스 문제
```

3단계: Proxy 설정 확인
```bash
# Nginx 설정에서 upstream 정의 확인
docker compose exec proxy cat /etc/nginx/conf.d/default.conf | grep -A 5 "upstream"

# 예상:
# upstream rest {
#   server rest:3000;
# }

# location /rest/ {
#   proxy_pass http://rest;
# }
```

4단계: 메모리/리소스 부족 여부 확인
```bash
# 호스트 리소스 사용률 확인
docker stats --no-stream

# 특정 서비스의 메모리 한계 도달 여부
docker inspect [service-name] | grep -E "Memory|OomKill"
```

5단계: 서비스 재시작
```bash
# 영향받은 백엔드 서비스만 재시작
docker compose restart rest auth

# 또는 전체 재시작
docker compose down
docker compose up -d
```

#### 3-4. DNS 전파 안됨

**증상**
```
도메인 등록 후 수 시간이 지났지만:
- nslookup / dig 에서 조회 실패
- 브라우저에서 접속 불가
- DNS 확인 사이트에서 일부 지역만 전파됨
```

**원인**
- DNS 레코드 설정 오류
- TTL(Time To Live)이 높아서 전파 지연
- 도메인 등록 기관과 네임서버 설정 불일치
- 글로벌 DNS 캐시

**해결 방법**

1단계: 도메인 등록 기관에서 네임서버 확인
```
GoDaddy, Namecheap 등에 로그인:
1. 도메인 관리 페이지 이동
2. "Nameservers" 또는 "DNS" 섹션 확인
3. 네임서버가 호스팅 제공자의 것으로 설정되어 있는지 확인

예:
✅ ns1.example-host.com, ns2.example-host.com
❌ 이전 호스팅의 네임서버가 남아있음
```

2단계: 호스팅 제공자의 DNS 컨트롤 패널 확인
```
AWS Route 53, Cloudflare, Digital Ocean 등:
1. DNS Management 페이지 이동
2. A 레코드 확인
3. 호스트: @ (또는 공백) → IP: 당신의 서버 IP

예:
✅ Name: @ (또는 your-domain.com)
   Type: A
   Value: 123.45.67.89 (당신의 IP)
   TTL: 300 (전파 속도, 낮을수록 빠름)
```

3단계: DNS 전파 진행 상황 모니터링
```bash
# 1시간마다 재확인
for i in {1..6}; do
  echo "Check $i at $(date)"
  nslookup your-domain.com 8.8.8.8
  sleep 600  # 10분 대기
done

# 또는 온라인 도구 사용
# https://whatsmydns.net/ 에서 도메인 입력
# 세계 각지의 DNS 서버에서 조회 결과 표시
```

4단계: TTL 감소 (다음 변경을 위해)
```
다음 DNS 변경이 필요하면:
1. 현재 DNS 레코드의 TTL을 3600 (1시간)으로 변경
2. 24시간 대기
3. 변경 적용
4. 변경 후 TTL을 86400 (24시간)으로 복구

이렇게 하면 다음 변경이 더 빨리 전파됨
```

5단계: 로컬 DNS 캐시 초기화 (테스트용)
```bash
# Linux
sudo systemctl restart systemd-resolved

# macOS
sudo dscacheutil -flushcache
sudo killall -HUP mDNSResponder

# Windows (관리자 권한)
ipconfig /flushdns
```

### 4. 데이터베이스 관련 오류

#### 4-1. PostgreSQL 연결 거부

**증상**
```
다른 서비스에서:
"could not connect to server: Connection refused"
또는
"FATAL: no pg_hba.conf entry"
```

**원인**
- PostgreSQL 서비스가 시작되지 않음
- DATABASE_URL이 잘못됨
- PostgreSQL 내부 포트(5432)가 수신 중이 아님
- 암호 또는 사용자명 오류
- pg_hba.conf 허용 설정 누락

**해결 방법**

1단계: PostgreSQL 컨테이너 상태 확인
```bash
docker compose ps postgres

# 상태가 "Up"이어야 함
# "Exited"이면 로그 확인
docker compose logs postgres --tail=100
```

2단계: PostgreSQL 내부 상태 확인
```bash
# PostgreSQL이 정말 수신 중인지 확인
docker compose exec postgres pg_isready -U postgres

# 출력:
# ✅ "accepting connections" = 정상
# ❌ "no response" = 문제
```

3단계: DATABASE_URL 확인
```bash
# .env 파일에서 확인
grep "DATABASE_URL" .env

# 형식:
# postgresql://user:password@host:5432/database?sslmode=require

# 각 부분 확인:
# - user: postgres (기본값)
# - password: .env 의 POSTGRES_PASSWORD 와 일치?
# - host: postgres (서비스명) - 또는 localhost (외부 접근 시)
# - port: 5432 (PostgreSQL 기본 포트)
# - database: postgres (기본값, 또는 supabase)
```

4단계: 다른 서비스에서 연결 테스트
```bash
# REST API 서비스에서 데이터베이스 연결 테스트
docker compose exec rest \
  psql "postgresql://postgres:your_password@postgres:5432/postgres" \
  -c "SELECT version();"

# 성공: PostgreSQL 버전 정보 출력
# 실패: 연결 오류 메시지
```

5단계: pg_hba.conf 설정 확인 (고급)
```bash
# PostgreSQL 설정 파일 확인
docker compose exec postgres cat /var/lib/postgresql/data/pg_hba.conf | head -20

# IPv4 연결이 허용되어 있는지 확인
# 예상:
# host    all             all             127.0.0.1/32            md5
# host    all             all             172.17.0.0/16           md5  (Docker 네트워크)
```

#### 4-2. 마이그레이션 실패

**증상**
```
마이그레이션 실행 시:
"migration failed" 또는
"ERROR: syntax error in SQL statement"
```

**원인**
- SQL 구문 오류
- 테이블/컬럼이 이미 존재
- 외래 키 제약 조건 위배
- 데이터베이스 권한 부족

**해결 방법**

1단계: 마이그레이션 파일 확인
```bash
# 마이그레이션 디렉토리 구조 확인
ls -la migrations/

# 최근 마이그레이션 파일 확인
ls -lt migrations/ | head -5
```

2단계: 마이그레이션 파일의 SQL 구문 확인
```bash
# 마지막으로 실패한 파일 열어보기
cat migrations/[failed_migration_file].sql

# 문법 오류 확인:
# ❌ CREATE TABLE users (... -- 세미콜론 누락
# ✅ CREATE TABLE users (...);
```

3단계: 이미 존재하는 테이블 확인
```bash
# PostgreSQL에 접속하여 테이블 목록 확인
docker compose exec postgres \
  psql -U postgres -c "\dt"

# 또는 특정 테이블만 확인
docker compose exec postgres \
  psql -U postgres -c "\d users"
```

4단계: 마이그레이션 순서 재확인
```bash
# Supabase는 타임스탐프 기반으로 마이그레이션 실행
# 파일명: YYYYMMDDHHMMSS_description.sql

# 파일 목록이 시간 순서대로 정렬되어 있는지 확인
ls migrations/ | sort

# 만약 순서가 뒤바뀌었으면 파일명 수정
```

5단계: 마이그레이션 롤백 및 재실행
```bash
# 실패한 마이그레이션 롤백
docker compose exec postgres \
  psql -U postgres -d postgres -c \
  "DELETE FROM _supabase_migrations WHERE name = '[failed_migration_name]';"

# 또는 데이터베이스 초기화 (⚠️ 모든 데이터 삭제)
docker compose exec postgres \
  psql -U postgres -c "DROP DATABASE IF EXISTS [database_name]; CREATE DATABASE [database_name];"

# 다시 마이그레이션 실행
docker compose exec [migration_service] supabase migration up
```

### 5. 일반 디버깅 체크리스트

프로젝트 배포 후 주기적으로 확인할 사항:

**주간 점검 (매주)**
- [ ] `docker compose ps` 에서 모든 서비스가 "Up (healthy)" 상태인지 확인
- [ ] `docker compose logs --tail=100` 에서 에러 메시지 스캔
- [ ] 로그 파일 크기 확인 및 로테이션 (`/var/log/docker/`)
- [ ] 메모리 사용률 확인 (`docker stats`)
- [ ] SSL 인증서 만료 여부 확인 (`docker compose logs certbot`)

**월간 점검 (매월)**
- [ ] 데이터베이스 백업 상태 확인
- [ ] .env 파일 민감정보 감사 (특히 KEY 값들이 강력한지)
- [ ] 방화벽 규칙 재확인
- [ ] 남은 리소스 용량 확인 (`df -h`)

**문제 발생 직후 점검**
- [ ] 정확한 에러 메시지 로그 수집
- [ ] 문제 발생 시간대 전후 로그 확인
- [ ] 최근 변경사항 확인 (.env, docker-compose.yml, DNS 등)
- [ ] 외부 의존성 상태 확인 (DNS, Let's Encrypt, 클라우드 서비스 등)

**유용한 명령어 모음**
```bash
# 모든 서비스 로그를 파일로 저장
docker compose logs > logs_backup_$(date +%Y%m%d_%H%M%S).txt

# 특정 문자열을 포함한 로그 검색
docker compose logs | grep -i "error\|warning\|failed"

# 컨테이너 환경 변수 확인
docker compose config | grep -A 50 "environment:"

# 불필요한 리소스 정리
docker system prune -a

# 특정 컨테이너 상세 정보 확인
docker inspect [container-name] | jq '.[] | {Name, State, Config}'
```
