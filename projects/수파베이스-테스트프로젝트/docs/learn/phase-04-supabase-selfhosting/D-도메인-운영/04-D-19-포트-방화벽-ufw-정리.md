## 개요

서버에 Supabase를 셀프호스팅할 때, 인터넷 연결을 위해 특정 포트를 열고 보안을 위해 나머지는 차단해야 합니다. 이 문서는 FE 개발자가 방화벽의 개념을 이해하고, Supabase 환경에서 안전하게 포트를 관리하는 방법을 배웁니다.

---

## 1. 방화벽(ufw)이 필요한 이유

### 1.1 FE 개발자 관점에서의 비유

```
브라우저의 CORS (Cross-Origin Resource Sharing)
  ↓ (비유)
서버의 ufw (Uncomplicated Firewall)
```

- **CORS**: 브라우저가 다른 도메인의 요청을 언제 허용할지 제어
- **ufw**: 서버가 외부 어떤 포트로부터의 연결을 언제 허용할지 제어

CORS 없이는 브라우저가 요청을 막고, 방화벽 없이는 누구나 서버의 모든 포트에 접근할 수 있습니다.

### 1.2 왜 필요한가?

- **보안**: 필요한 포트만 열고 나머지는 차단 → 악의적인 접근 방지
- **네트워크 위생**: 의도하지 않은 서비스 노출 방지
- **규정 준수**: 기업의 보안 정책 충족

## 2. ufw 기본 명령어

### 2.1 ufw 설치 및 활성화

대부분의 Ubuntu 서버에는 ufw가 이미 설치되어 있습니다.

```bash
# ufw 설치 (필요한 경우)
sudo apt-get install ufw

# ufw 활성화
sudo ufw enable

# 활성화 확인
sudo ufw status verbose
```

### 2.2 포트 열기 (allow)

```bash
# TCP 포트 열기
sudo ufw allow 22/tcp

# UDP 포트 열기
sudo ufw allow 53/udp

# TCP와 UDP 모두 열기 (포트 번호만 지정)
sudo ufw allow 80

# 포트 범위 열기
sudo ufw allow 1000:2000/tcp
```

### 2.3 포트 닫기 (deny)

```bash
# 포트 차단
sudo ufw deny 5432

# 이미 열려있는 포트 제거
sudo ufw delete allow 5432
```

### 2.4 규칙 확인 및 관리

```bash
# 현재 ufw 상태 확인
sudo ufw status

# 상세 정보 포함해서 확인
sudo ufw status verbose

# 규칙 번호 포함해서 확인
sudo ufw status numbered

# 특정 규칙 삭제
sudo ufw delete allow 3000
```

## 3. Supabase 셀프호스팅에서 필요한 포트

### 3.1 반드시 열어야 할 포트

| 포트 | 프로토콜 | 목적 | 설명 |
|------|---------|------|------|
| **22** | TCP | SSH | 서버 접속 및 관리 |
| **80** | TCP | HTTP | 웹 사이트 접근 (Caddy가 리다이렉트) |
| **443** | TCP | HTTPS | 보안 웹 연결 (Caddy 포트) |

### 3.2 설정 명령어

```bash
# SSH 포트 열기
sudo ufw allow 22/tcp

# HTTP 포트 열기
sudo ufw allow 80/tcp

# HTTPS 포트 열기
sudo ufw allow 443/tcp

# 한 번에 모두 열기
sudo ufw allow 22/tcp && sudo ufw allow 80/tcp && sudo ufw allow 443/tcp
```

## 4. 절대 열면 안 되는 포트

### 4.1 위험한 포트 목록

| 포트 | 서비스 | 왜 위험한가? |
|------|-------|-----------|
| **5432** | PostgreSQL 데이터베이스 | 직접 DB 접근 → 모든 데이터 노출 |
| **8000** | Kong API Gateway (직접) | 인증 우회 가능 |
| **3000** | Supabase Studio (직접) | 어드민 패널 직접 노출 |
| **6379** | Redis | 캐시 데이터 직접 노출 |
| **9000** | Realtime (직접) | 실시간 데이터 스트림 노출 |

### 4.2 이 포트들을 절대 열면 안 되는 이유

```
외부 사용자가 5432(PostgreSQL)에 직접 접근 가능

→ 사용자 인증 없이 모든 데이터 접근 가능
→ 데이터 탈취, 삭제, 변조 가능
→ 보안 대재앙
```

---

## 5. Caddy가 앞에 있으므로 80/443만 열면 되는 이유

### 5.1 아키텍처 이해

```
인터넷 사용자
    ↓ (80, 443 포트)
Caddy (리버스 프록시)
    ↓ (내부 통신)
Kong (8000) → Supabase API
PostgreSQL (5432)
Realtime (9000)
```

### 5.2 왜 80/443만 열어도 되는가?

1. **Caddy는 리버스 프록시**: 외부의 HTTPS 요청을 받아서 내부 서비스로 라우팅
2. **내부 통신은 안전**: 내부 네트워크는 방화벽이 관여하지 않음
3. **인증 레이어**: 모든 요청이 Kong을 통해 인증됨
4. **외부 노출 최소화**: 필요한 포트만 노출

### 5.3 포트 매핑 예시

```
사용자 HTTPS 요청 (443)
  ↓
Caddy (443)
  ↓ (내부 포워딩)
Kong (8000) - 외부에서 직접 접근 불가
  ↓
Supabase API
```

## 6. Docker와 ufw의 충돌 문제

### 6.1 문제 상황

Docker를 사용하면 ufw 규칙을 우회할 수 있습니다.

```
문제: Docker가 iptables를 직접 조작하기 때문에
      ufw의 규칙을 무시하고 포트를 노출시킬 수 있음
```

### 6.2 원인

- **ufw**: `ufw allow` 규칙은 iptables 규칙을 생성
- **Docker**: 자체 iptables 규칙을 추가 (ufw와 별개로)
- **결과**: Docker 포트가 ufw를 우회해서 외부에 노출

### 6.3 해결 방법

#### 방법 1: Docker daemon.json 수정 (권장)

```bash
# Docker 설정 파일 열기
sudo nano /etc/docker/daemon.json
```

다음 내용을 추가합니다.

```json
{
  "iptables": false
}
```

설정을 적용합니다.

```bash
# Docker 데몬 재시작
sudo systemctl restart docker
```

#### 방법 2: UFW-DOCKER 규칙 적용

Docker 이미지를 사용하여 UFW와 호환되는 규칙 설정:

```bash
# UFW Docker 통합 스크립트 다운로드
sudo curl -o /etc/ufw/after.rules https://raw.githubusercontent.com/chaifeng/ufw-docker/master/ufw-docker.sh

# 다운로드한 스크립트 실행
sudo bash /etc/ufw/after.rules
```

### 6.4 확인 방법

Docker 컨테이너를 띄운 후, 포트가 실제로 노출되는지 확인합니다.

```bash
# 외부에서 접근 가능한 포트 확인
sudo netstat -tulpn | grep LISTEN

# 또는 ss 명령어
sudo ss -tulpn | grep LISTEN
```

## 7. 최종 포트 정리 표

### Supabase 셀프호스팅 - 포트 관리 완벽 가이드

| 포트 | 프로토콜 | 서비스 | 외부 노출 | ufw 설정 | 비고 |
|------|---------|-------|---------|---------|------|
| **22** | TCP | SSH | ✅ 필수 | `allow` | 서버 관리용 |

| **5432** | TCP | PostgreSQL | ❌ 금지 | `deny` | 절대 외부 노출 금지 |
| **6379** | TCP | Redis | ❌ 금지 | `deny` | 캐시 데이터 보호 |
| **8000** | TCP | Kong | ❌ 금지 | `deny` | Caddy를 통해서만 접근 |
| **9000** | TCP | Realtime | ❌ 금지 | `deny` | Caddy를 통해서만 접근 |

## 8. 실습: 포트 설정 완료하기

### 8.1 현재 상태 확인

```bash
# 현재 방화벽 상태 확인
sudo ufw status verbose
```

### 8.2 필수 포트 열기

```bash
# SSH, HTTP, HTTPS 포트 열기
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp

```

### 8.3 위험한 포트 명시적으로 차단 (선택사항)

```bash
# 데이터베이스 포트 차단
sudo ufw deny 5432/tcp

# Kong 직접 접근 차단
sudo ufw deny 8000/tcp

# Studio 직접 접근 차단
sudo ufw deny 3000/tcp

# Realtime 직접 접근 차단
sudo ufw deny 9000/tcp

# Redis 직접 접근 차단
sudo ufw deny 6379/tcp
```

### 8.4 설정 확인

```bash
# 규칙 번호와 함께 확인
sudo ufw status numbered

# 상세 정보 확인

```

## 9. Docker와 ufw 충돌 해결하기

### 9.1 Docker daemon.json 수정

```bash
# 설정 파일 확인
sudo cat /etc/docker/daemon.json

# 없으면 생성, 있으면 편집
sudo nano /etc/docker/daemon.json
```

다음을 추가합니다.

```json
{
  "iptables": false
}
```

### 9.2 Docker 재시작

```bash
# Docker 데몬 재시작
sudo systemctl restart docker

# 상태 확인
sudo systemctl status docker
```

### 9.3 검증

```bash
# Supabase 컨테이너가 실행 중인지 확인
docker ps

# 노출된 포트 확인 (5432, 3000, 8000 등이 보이면 위험)
docker ps --format "table {{.Names}}\t{{.Ports}}"
```

포트 목록에 `0.0.0.0:5432->5432/tcp` 같은 형태가 없어야 안전합니다.

## 10. 자주 겪는 오류 및 해결책

### 10.1 "SSH 접속이 안 된다"

**증상**: `ssh user@server.com` 시도 시 timeout 또는 connection refused

**원인**:
- ufw에서 22번 포트를 차단했거나
- ufw가 활성화되지 않았거나
- 서버의 SSH 서비스가 실행되지 않음

**해결책**:

```bash
# SSH 포트 허용 (서버 물리 접근 필요)
sudo ufw allow 22/tcp

# ufw 상태 확인
sudo ufw status

# SSH 서비스 재시작
sudo systemctl restart ssh
```

### 10.2 "웹 사이트에 접속할 수 없다"

**증상**: `https://yourdomain.com` 접속 불가

**원인**:
- 80/443 포트가 차단됨
- Caddy가 실행되지 않음
- Docker가 ufw를 우회해서 내부에서만 접근 가능

**해결책**:

```bash
# HTTP/HTTPS 포트 확인
sudo ufw status | grep -E "(80|443)"

# 포트가 DENY인 경우 ALLOW로 변경
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# Caddy 상태 확인
docker ps | grep caddy
# 또는
sudo systemctl status caddy

# Docker 충돌 확인
docker port <caddy_container_id>
```

### 10.3 "Django/Flask 앱이 직접 접근 가능하다"

**증상**: `http://server.com:8000` 으로 직접 앱 접근 가능 (위험!)

**원인**:
- Docker가 iptables를 직접 조작하여 ufw를 우회
- `/etc/docker/daemon.json`에서 `iptables` 설정이 true

**해결책**:

```bash
# Docker 설정 수정
sudo nano /etc/docker/daemon.json

# "iptables": false 추가
# 예:
# {
#   "iptables": false
# }

# Docker 재시작
sudo systemctl restart docker

# 컨테이너 재시작 (선택)
docker restart <container_id>

# 포트 재확인
docker ps
```

### 10.4 "PostgreSQL에 외부에서 접근 가능하다"

**증상**: `psql -h yourdomain.com -U postgres` 로 연결 가능 (대재앙!)

**원인**:
- 5432 포트가 ALLOW로 설정됨
- Docker가 5432를 외부에 노출

**긴급 조치**:

```bash
# 즉시 5432 포트 차단
sudo ufw deny 5432/tcp

# 방화벽 재로드
sudo ufw reload

# Docker 컨테이너의 포트 바인딩 확인
docker ps

# 컨테이너 실행 시 포트 바인딩 제거해야 함
# 예: docker-compose.yml에서
# - "5432:5432" 제거 → "127.0.0.1:5432:5432" 변경
```

## 11. 체크리스트

학습을 완료하기 전에 다음을 확인하세요.

```
☐ ufw가 활성화되어 있다 (sudo ufw status)
☐ 22, 80, 443 포트가 ALLOW 상태다
☐ 5432, 8000, 3000, 6379, 9000이 차단되어 있다
☐ Docker daemon.json에서 "iptables": false 설정했다
☐ Docker를 재시작했다
☐ 웹사이트에 HTTPS로 접속 가능하다
☐ SSH로 접속 가능하다
☐ 데이터베이스 포트(5432)가 외부에서 접근 불가능하다

검증 명령어:
sudo ufw status verbose
docker ps
docker port <container_id>
```
